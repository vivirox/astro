Solid is a declarative Java\+Script library for creating user interfaces. Instead of using a Virtual DOM, it compiles its templates to real DOM nodes and updates them with fine-\/grained reactions. Declare your state and use it throughout your app, and when a piece of state changes, only the code that depends on it will rerun. Check out our \href{https://www.youtube.com/watch?v=cELFZQAMdhQ}{\texttt{ intro video}} or read on!\hypertarget{README.md_autotoc_md30955}{}\doxysubsubsection{\texorpdfstring{Key Features}{Key Features}}\label{README.md_autotoc_md30955}

\begin{DoxyItemize}
\item Fine-\/grained updates to the real DOM
\item Declarative data\+: model your state as a system with reactive primitives
\item Render-\/once mental model\+: your components are regular Java\+Script functions that run once to set up your view
\item Automatic dependency tracking\+: accessing your reactive state subscribes to it
\item \href{https://dev.to/this-is-learning/javascript-framework-todomvc-size-comparison-504f}{\texttt{ Small}} and \href{https://krausest.github.io/js-framework-benchmark/current.html}{\texttt{ fast}}
\item Simple\+: learn a few powerful concepts that can be reused, combined, and built on top of
\item Provides modern framework features like JSX, fragments, Context, Portals, Suspense, streaming SSR, progressive hydration, Error Boundaries and concurrent rendering.
\item Naturally debuggable\+: A {\ttfamily \texorpdfstring{$<$}{<}div\texorpdfstring{$>$}{>}} is a real div, so you can use your browser\textquotesingle{}s devtools to inspect the rendering
\item \href{https://github.com/solidjs/solid/tree/main/packages/solid-element\#readme}{\texttt{ Web component friendly}} and can author custom elements
\item Isomorphic\+: render your components on the client and the server
\item Universal\+: write \href{https://github.com/solidjs/solid/releases/tag/v1.2.0}{\texttt{ custom renderers}} to use Solid anywhere
\item A growing community and ecosystem with active core team support
\end{DoxyItemize}

\hypertarget{README.md_autotoc_md30956}{}\doxysubsubsection{\texorpdfstring{Why Solid?}{Why Solid?}}\label{README.md_autotoc_md30956}
\hypertarget{README.md_autotoc_md30957}{}\doxysubsubsubsection{\texorpdfstring{Performant}{Performant}}\label{README.md_autotoc_md30957}
Meticulously engineered for performance and with half a decade of research behind it, Solid\textquotesingle{}s performance is almost indistinguishable from optimized vanilla Java\+Script (See Solid on the \href{https://krausest.github.io/js-framework-benchmark/current.html}{\texttt{ JS Framework Benchmark}}). Solid is \href{https://bundlephobia.com/package/solid-js@1.3.15}{\texttt{ small}} and completely tree-\/shakable, and \href{https://levelup.gitconnected.com/how-we-wrote-the-fastest-javascript-ui-framework-again-db097ddd99b6}{\texttt{ fast}} when rendering on the server, too. Whether you\textquotesingle{}re writing a fully client-\/rendered SPA or a server-\/rendered app, your users see it faster than ever. (\href{https://dev.to/ryansolid/thinking-granular-how-is-solidjs-so-performant-4g37}{\texttt{ Read more about Solid\textquotesingle{}s performance}} from the library\textquotesingle{}s creator.)\hypertarget{README.md_autotoc_md30958}{}\doxysubsubsubsection{\texorpdfstring{Powerful}{Powerful}}\label{README.md_autotoc_md30958}
Solid is fully-\/featured with everything you can expect from a modern framework. Performant state management is built-\/in with Context and Stores\+: you don\textquotesingle{}t have to reach for a third party library to manage global state (if you don\textquotesingle{}t want to). With Resources, you can use data loaded from the server like any other piece of state and build a responsive UI for it thanks to Suspense and concurrent rendering. And when you\textquotesingle{}re ready to move to the server, Solid has full SSR and serverless support, with streaming and progressive hydration to get to interactive as quickly as possible. (Check out our full \href{https://www.solidjs.com/tutorial/introduction_basics}{\texttt{ interactive features walkthrough}}.)\hypertarget{README.md_autotoc_md30959}{}\doxysubsubsubsection{\texorpdfstring{Pragmatic}{Pragmatic}}\label{README.md_autotoc_md30959}
Do more with less\+: use simple, composable primitives without hidden rules and gotchas. In Solid, components are just functions -\/ rendering is determined purely by how your state is used -\/ so you\textquotesingle{}re free to organize your code how you like and you don\textquotesingle{}t have to learn a new rendering system. Solid encourages patterns like declarative code and read-\/write segregation that help keep your project maintainable, but isn\textquotesingle{}t opinionated enough to get in your way.\hypertarget{README.md_autotoc_md30960}{}\doxysubsubsubsection{\texorpdfstring{Productive}{Productive}}\label{README.md_autotoc_md30960}
Solid is built on established tools like JSX and Type\+Script and integrates with the Vite ecosystem. Solid\textquotesingle{}s bare-\/metal, minimal abstractions give you direct access to the DOM, making it easy to use your favorite native Java\+Script libraries like D3. And the Solid ecosystem is growing fast, with \href{https://github.com/solidjs-community/solid-primitives}{\texttt{ custom primitives}}, \href{https://github.com/hope-ui/hope-ui}{\texttt{ component libraries}}, and build-\/time utilities that let you \href{https://github.com/LXSMNSYC/solid-labels}{\texttt{ write Solid code in new ways}}.
