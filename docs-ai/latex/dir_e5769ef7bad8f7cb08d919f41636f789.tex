\begin{DoxyItemize}
\item {\bfseries{{\ttfamily vue(.runtime).global(.prod).js}}}\+:
\begin{DoxyItemize}
\item For direct use via {\ttfamily \texorpdfstring{$<$}{<}script src="{}..."{}\texorpdfstring{$>$}{>}} in the browser. Exposes the {\ttfamily Vue} global.
\item Note that global builds are not \href{https://github.com/umdjs/umd}{\texttt{ UMD}} builds. They are built as \href{https://developer.mozilla.org/en-US/docs/Glossary/IIFE}{\texttt{ IIFEs}} and is only meant for direct use via {\ttfamily \texorpdfstring{$<$}{<}script src="{}..."{}\texorpdfstring{$>$}{>}}.
\item In-\/browser template compilation\+:
\begin{DoxyItemize}
\item {\bfseries{{\ttfamily vue.\+global.\+js}}} is the "{}full"{} build that includes both the compiler and the runtime so it supports compiling templates on the fly.
\item {\bfseries{{\ttfamily vue.\+runtime.\+global.\+js}}} contains only the runtime and requires templates to be pre-\/compiled during a build step.
\end{DoxyItemize}
\item Inlines all Vue core internal packages -\/ i.\+e. it\textquotesingle{}s a single file with no dependencies on other files. This means you {\bfseries{must}} import everything from this file and this file only to ensure you are getting the same instance of code.
\item Contains hard-\/coded prod/dev branches, and the prod build is pre-\/minified. Use the {\ttfamily \texorpdfstring{$\ast$}{*}.prod.\+js} files for production.
\end{DoxyItemize}
\item {\bfseries{{\ttfamily vue(.runtime).esm-\/browser(.prod).js}}}\+:
\begin{DoxyItemize}
\item For usage via native ES modules imports (in browser via {\ttfamily \texorpdfstring{$<$}{<}script type="{}module"{}\texorpdfstring{$>$}{>}}).
\item Shares the same runtime compilation, dependency inlining and hard-\/coded prod/dev behavior with the global build.
\end{DoxyItemize}

\begin{DoxyItemize}
\item {\bfseries{{\ttfamily vue(.runtime).esm-\/bundler.\+js}}}\+:
\begin{DoxyItemize}
\item For use with bundlers like {\ttfamily webpack}, {\ttfamily rollup} and {\ttfamily parcel}.
\item Leaves prod/dev branches with {\ttfamily process.\+env.\+NODE\+\_\+\+ENV} guards (must be replaced by bundler)
\item Does not ship minified builds (to be done together with the rest of the code after bundling)
\item Imports dependencies (e.\+g. {\ttfamily @vue/runtime-\/core}, {\ttfamily @vue/compiler-\/core})
\begin{DoxyItemize}
\item Imported dependencies are also {\ttfamily esm-\/bundler} builds and will in turn import their dependencies (e.\+g. {\ttfamily @vue/runtime-\/core} imports {\ttfamily @vue/reactivity})
\item This means you {\bfseries{can}} install/import these deps individually without ending up with different instances of these dependencies, but you must make sure they all resolve to the same version.
\end{DoxyItemize}
\item In-\/browser template compilation\+:
\begin{DoxyItemize}
\item {\bfseries{{\ttfamily vue.\+runtime.\+esm-\/bundler.\+js} (default)}} is runtime only, and requires all templates to be pre-\/compiled. This is the default entry for bundlers (via {\ttfamily module} field in {\ttfamily package.\+json}) because when using a bundler templates are typically pre-\/compiled (e.\+g. in {\ttfamily \texorpdfstring{$\ast$}{*}.vue} files).
\item {\bfseries{{\ttfamily vue.\+esm-\/bundler.\+js}}}\+: includes the runtime compiler. Use this if you are using a bundler but still want runtime template compilation (e.\+g. in-\/\+DOM templates or templates via inline Java\+Script strings). You will need to configure your bundler to alias {\ttfamily vue} to this file.
\end{DoxyItemize}
\end{DoxyItemize}
