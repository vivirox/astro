For \href{https://docs.microsoft.com/en-us/archive/blogs/larryosterman/why-is-the-dos-path-character}{\texttt{ historical reasons}}, windows followed MS-\/\+DOS and used backslash for separating paths rather than slash used for mac\+OS, Linux, and other Posix operating systems. Nowadays, \href{https://docs.microsoft.com/en-us/windows/win32/fileio/naming-a-file?redirectedfrom=MSDN}{\texttt{ Windows}} supports both Slash and Backslash for paths. \href{https://nodejs.org/api/path.html}{\texttt{ Node.\+js\textquotesingle{}s built-\/in {\ttfamily path} module}} in the default operation of the path module varies based on the operating system on which a Node.\+js application is running. Specifically, when running on a Windows operating system, the path module will assume that Windows-\/style paths are being used. {\bfseries{This makes inconsistent code behavior between Windows and POSIX.}}

Compared to popular \href{https://github.com/anodynos/upath}{\texttt{ upath}}, pathe provides {\bfseries{identical exports}} of Node.\+js with normalization on {\bfseries{all operations}} and is written in modern {\bfseries{ESM/\+Type\+Script}} and has {\bfseries{no dependency on Node.\+js}}!

This package is a drop-\/in replacement of the Node.\+js\textquotesingle{}s \href{https://nodejs.org/api/path.html}{\texttt{ path module}} module and ensures paths are normalized with slash {\ttfamily /} and work in environments including Node.\+js.\hypertarget{README.md_autotoc_md25857}{}\doxysubsection{\texorpdfstring{üíø Usage}{üíø Usage}}\label{README.md_autotoc_md25857}
