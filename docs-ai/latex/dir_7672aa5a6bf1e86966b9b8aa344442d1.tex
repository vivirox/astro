{\bfseries{unified}} is an interface for processing text using syntax trees. Syntax trees are a representation of text understandable to programs. Those programs, called \texorpdfstring{$\ast$}{*}plugin\texorpdfstring{$\ast$}{*}s, take these trees and inspect and modify them. To get to the syntax tree from text, there is a \texorpdfstring{$\ast$}{*}parser\texorpdfstring{$\ast$}{*}. To get from that back to text, there is a \texorpdfstring{$\ast$}{*}compiler\texorpdfstring{$\ast$}{*}. This is the \texorpdfstring{$\ast$}{*}process\texorpdfstring{$\ast$}{*} of a {\itshape processor}.

\hypertarget{readme.md_autotoc_md413}{}\doxysubsubsubsubsubsection{\texorpdfstring{Processors}{Processors}}\label{readme.md_autotoc_md413}
Every {\bfseries{processor}} implements another processor. To create a processor, call another processor. The new processor is configured to work the same as its ancestor. But when the descendant processor is configured in the future it does not affect the ancestral processor.

When processors are exposed from a module (for example, {\ttfamily unified} itself) they should not be configured directly, as that would change their behavior for all module users. Those processors are \texorpdfstring{$\ast$}{*}frozen\texorpdfstring{$\ast$}{*} and they should be called to create a new processor before they are used.\hypertarget{readme.md_autotoc_md414}{}\doxysubsubsubsubsubsection{\texorpdfstring{Syntax trees}{Syntax trees}}\label{readme.md_autotoc_md414}
The {\bfseries{syntax trees}} used in {\bfseries{unified}} are \href{https://github.com/syntax-tree/unist}{\texttt{ {\bfseries{unist}}}} nodes. A \href{https://github.com/syntax-tree/unist\#node}{\texttt{ {\bfseries{node}}}} is a plain Java\+Script objects with a {\ttfamily type} field. The semantics of nodes and format of syntax trees is defined by other projects.

There are several \href{https://github.com/syntax-tree/unist\#list-of-utilities}{\texttt{ {\itshape utilities}}} for working with nodes.

\hypertarget{readme.md_autotoc_md421}{}\doxysubsubsubsubsubsection{\texorpdfstring{Processing between syntaxes}{Processing between syntaxes}}\label{readme.md_autotoc_md421}
\texorpdfstring{$\ast$}{*}\+Processors\texorpdfstring{$\ast$}{*} can be combined in two modes.

{\bfseries{Bridge}} mode transforms the \texorpdfstring{$\ast$}{*}syntax tree\texorpdfstring{$\ast$}{*} from one format ({\itshape origin}) to another ({\itshape destination}). Another processor runs on the destination tree. Finally, the original processor continues transforming the origin tree.

{\bfseries{Mutate}} mode also transforms the syntax tree from one format to another. But the original processor continues transforming the destination tree.

In the previous example (‚Äú\+Programming interface‚Äù), {\ttfamily remark-\/retext} is used in {\itshape bridge} mode\+: the origin syntax tree is kept after \href{https://github.com/retextjs/retext}{\texttt{ {\bfseries{retext}}}} is done; whereas {\ttfamily remark-\/rehype} is used in {\itshape mutate} mode\+: it sets a new syntax tree and discards the origin tree.
