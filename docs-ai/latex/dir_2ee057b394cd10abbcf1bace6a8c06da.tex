This package is a \href{https://github.com/unifiedjs/unified}{\texttt{ unified}} (\href{https://github.com/remarkjs/remark}{\texttt{ remark}}) plugin to enable the extensions to markdown that Git\+Hub adds with GFM\+: autolink literals ({\ttfamily www.\+x.\+com}), footnotes ({\ttfamily \mbox{[}\texorpdfstring{$^\wedge$}{\string^}1\mbox{]}}), strikethrough ({\ttfamily \texorpdfstring{$\sim$}{\string~}\texorpdfstring{$\sim$}{\string~}stuff\texorpdfstring{$\sim$}{\string~}\texorpdfstring{$\sim$}{\string~}}), tables ({\ttfamily \texorpdfstring{$\vert$}{|} cell \texorpdfstring{$\vert$}{|}‚Ä¶}), and tasklists ({\ttfamily \texorpdfstring{$\ast$}{*} \mbox{[}x\mbox{]}}). You can use this plugin to add support for parsing and serializing them. These extensions by Git\+Hub to Common\+Mark are called \href{https://github.github.com/gfm/}{\texttt{ GFM}} (Git\+Hub Flavored Markdown).

This plugin does not handle how markdown is turned to HTML. That‚Äôs done by \href{https://github.com/remarkjs/remark-rehype}{\texttt{ {\ttfamily remark-\/rehype}}}. If your content is not in English and uses footnotes, you should configure that plugin. When generating HTML, you might also want to enable \href{https://github.com/rehypejs/rehype-slug}{\texttt{ {\ttfamily rehype-\/slug}}} to add {\ttfamily id}s on headings.

A different plugin, \href{https://github.com/remarkjs/remark-frontmatter}{\texttt{ {\ttfamily remark-\/frontmatter}}}, adds support for frontmatter. Git\+Hub supports YAML frontmatter for files in repos and Gists but they don‚Äôt treat it as part of GFM.

Another plugin, \href{https://github.com/remarkjs/remark-github}{\texttt{ {\ttfamily remark-\/github}}}, adds support for how markdown works in relation to a certain Git\+Hub repo in comments, issues, PRs, and releases, by linking references to commits, issues, and users.

Yet another plugin, \href{https://github.com/remarkjs/remark-breaks}{\texttt{ {\ttfamily remark-\/breaks}}}, turns soft line endings (enters) into hard breaks ({\ttfamily \texorpdfstring{$<$}{<}br\texorpdfstring{$>$}{>}}s). Git\+Hub does this in a few places (comments, issues, PRs, and releases).\hypertarget{readme.md_autotoc_md29127}{}\doxysubsection{\texorpdfstring{When should I use this?}{When should I use this?}}\label{readme.md_autotoc_md29127}
This project is useful when you want to support the same features that Git\+Hub does in files in a repo, Gists, and several other places. Users frequently believe that some of these extensions, specifically autolink literals and tables, are part of normal markdown, so using {\ttfamily remark-\/gfm} will help match your implementation to their understanding of markdown. There are several edge cases where Git\+Hub‚Äôs implementation works in unexpected ways or even different than described in their spec, so {\itshape writing} in GFM is not always the best choice.

If you {\itshape just} want to turn markdown into HTML (with maybe a few extensions such as GFM), we recommend \href{https://github.com/micromark/micromark}{\texttt{ {\ttfamily micromark}}} with \href{https://github.com/micromark/micromark-extension-gfm}{\texttt{ {\ttfamily micromark-\/extension-\/gfm}}} instead. If you don‚Äôt use plugins and want to access the syntax tree, you can use \href{https://github.com/syntax-tree/mdast-util-from-markdown}{\texttt{ {\ttfamily mdast-\/util-\/from-\/markdown}}} with \href{https://github.com/syntax-tree/mdast-util-gfm}{\texttt{ {\ttfamily mdast-\/util-\/gfm}}}.\hypertarget{readme.md_autotoc_md29128}{}\doxysubsection{\texorpdfstring{Install}{Install}}\label{readme.md_autotoc_md29128}
