An {\bfseries{\href{https://github.com/kkos/oniguruma}{\texttt{ Oniguruma}} to Java\+Script regex translator}} that runs in the browser and on your server. Use it to\+:


\begin{DoxyItemize}
\item Take advantage of Oniguruma\textquotesingle{}s many extended regex features in Java\+Script.
\item Run regexes written for Oniguruma from Java\+Script, such as those used in Text\+Mate grammars (used by VS Code, Git\+Hub, \href{https://shiki.style/}{\texttt{ Shiki}}, etc.).
\item Share regexes across your Ruby\textsuperscript{‚ú≥Ô∏é} or PHP ({\ttfamily mb\+\_\+ereg}, etc.) and Java\+Script code.
\end{DoxyItemize}

Compared to running the Oniguruma C library via WASM bindings using \href{https://github.com/microsoft/vscode-oniguruma}{\texttt{ vscode-\/oniguruma}}, this library is \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\sim$}{\string~}4\% of the size\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} and its regexes often run much faster since they run as native Java\+Script.

\begin{quote}
You can further reduce bundle size by precompiling your regexes. In many cases, that avoids the need for any runtime dependency. Some regex conversions rely on advanced, subclass-\/based emulation, in which case the tree-\/shakable {\ttfamily Emulated\+Reg\+Exp} class (3 kB minzip) is still needed after precompilation. \end{quote}


Oniguruma-\/\+To-\/\+ES deeply understands the hundreds of large and small differences between Oniguruma and Java\+Script regex syntax and behavior, across multiple Java\+Script version targets. It\textquotesingle{}s {\itshape obsessive} about ensuring that the emulated features it supports have {\bfseries{exactly the same behavior}}, even in extreme edge cases. And it\textquotesingle{}s been battle-\/tested on tens of thousands of real-\/world Oniguruma regexes used in Text\+Mate grammars. It uses \href{https://github.com/slevithan/oniguruma-parser}{\texttt{ oniguruma-\/parser}} to build an AST, as part of the transpilation process.

 }\hypertarget{README.md_autotoc_md25192}{}\doxysubsection{\texorpdfstring{{\ttfamily „äóÔ∏è Unicode}}{{\ttfamily „äóÔ∏è Unicode}}}\label{README.md_autotoc_md25192}

{\ttfamily Oniguruma-\/\+To-\/\+ES fully supports mixed case-\/sensitivity (ex\+: {\ttfamily (?i)a(?-\/i)a}) and handles the Unicode edge cases regardless of Java\+Script target.}

{\ttfamily Oniguruma-\/\+To-\/\+ES focuses on being lightweight to make it better for use in browsers. This is partly achieved by not including heavyweight Unicode character data, which imposes a few minor/rare restrictions\+:}

{\ttfamily 
\begin{DoxyItemize}
\item Character class intersection and nested negated character classes are unsupported with target {\ttfamily ES2018}. Use target {\ttfamily ES2024} (supported by Node.\+js 20 and 2023-\/era browsers) or later if you need support for these features.
\item With targets before {\ttfamily ES2025}, a handful of Unicode properties that target a specific character case (ex\+: {\ttfamily \textbackslash{}p\{Lower\}}) can\textquotesingle{}t be used case-\/insensitively in patterns that contain other characters with a specific case that are used case-\/sensitively.
\begin{DoxyItemize}
\item In other words, almost every usage is fine, including {\ttfamily A\textbackslash{}p\{Lower\}}, {\ttfamily (?i)A\textbackslash{}p\{Lower\}}, {\ttfamily (?i\+:A)\textbackslash{}p\{Lower\}}, {\ttfamily (?i)A(?-\/i)\textbackslash{}p\{Lower\}}, and {\ttfamily \textbackslash{}w(?i)\textbackslash{}p\{Lower\}}, but not {\ttfamily A(?i)\textbackslash{}p\{Lower\}}.
\item Using these properties case-\/insensitively is basically never done intentionally, so you\textquotesingle{}re unlikely to encounter this error unless it\textquotesingle{}s catching a mistake.
\end{DoxyItemize}
\item Oniguruma-\/\+To-\/\+ES uses the version of Unicode supported natively by your Java\+Script environment. Using Unicode properties via {\ttfamily \textbackslash{}p\{‚Ä¶\}} that were added in a later version of Unicode than the environment supports results in a runtime error. This is an extreme edge case since modern Java\+Script environments support recent versions of Unicode.
\end{DoxyItemize}}
