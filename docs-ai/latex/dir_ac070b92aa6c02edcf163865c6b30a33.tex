A universal solution for loading configurations.\hypertarget{README.md_autotoc_md34111}{}\doxysubsection{\texorpdfstring{Why?}{Why?}}\label{README.md_autotoc_md34111}
Configuration is hard, especially when you want to build an ecosystem of your tools.

You want your tools to be general and easy to use, you allow your config to be defined in a custom field of {\ttfamily package.\+json}.

You want your tools to be easy to integrate, you allow the configs to be defined in other tools\textquotesingle{} configurations like {\ttfamily vite.\+config.\+js} or {\ttfamily webpack.\+config.\+js}.

You want the configs to be agnostic and probably need to be load by IDE, you create new config files like {\ttfamily .myconfigrc}.

You want the configs to also be flexible and dynamic, you make your config files a Java\+Script file, like {\ttfamily my.\+config.\+js}.

Then you want users to be able to use ESM and Type\+Script, you also make your config accepting {\ttfamily .ts} or {\ttfamily .mjs}.

So users\textquotesingle{} project root end up with a lot of config files like {\ttfamily .npmrc}, {\ttfamily rollup.\+config.\+js}, {\ttfamily .eslintrc}, {\ttfamily tsconfig.\+json}, {\ttfamily jest.\+config.\+js}, {\ttfamily postcss.\+config.\+js}, {\ttfamily nuxt.\+config.\+js}, {\ttfamily vite.\+config.\+cjs}, {\ttfamily windi.\+config.\+ts}, etc. And each of them use different syntax, in JSON, in CJS, in ESM, in Type\+Script, in INI, in TOML...

{\ttfamily unconfig} can\textquotesingle{}t solve this fragmentation entirely, but it\textquotesingle{}s trying to make loading them easier for tool authors.\hypertarget{README.md_autotoc_md34112}{}\doxysubsection{\texorpdfstring{Usage}{Usage}}\label{README.md_autotoc_md34112}
