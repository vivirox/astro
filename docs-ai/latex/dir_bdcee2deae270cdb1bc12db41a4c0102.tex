This package contains extensions that add support for the extra autolink syntax enabled by GFM to \href{https://github.com/micromark/micromark}{\texttt{ {\ttfamily micromark}}}.

Git\+Hub employs different algorithms to autolink\+: one at parse time and one at transform time (similar to how @mentions are done at transform time). This difference can be observed because character references and escapes are handled differently. But also because issues/\+PRs/comments omit (perhaps by accident?) the second algorithm for {\ttfamily www.}, {\ttfamily \href{http://}{\texttt{ http\+://}}}, and {\ttfamily \href{https://}{\texttt{ https\+://}}} links (but not for email links).

As this is a syntax extension, it focuses on the first algorithm. The second algorithm is performed by \href{https://github.com/syntax-tree/mdast-util-gfm-autolink-literal}{\texttt{ {\ttfamily mdast-\/util-\/gfm-\/autolink-\/literal}}}. The {\ttfamily html} part of this micromark extension does not operate on an AST and hence can’t perform the second algorithm.

The implementation of autolink literal on github.\+com is currently buggy. The bugs have been reported on \href{https://github.com/github/cmark-gfm}{\texttt{ {\ttfamily cmark-\/gfm}}}. This micromark extension matches github.\+com except for its bugs.\hypertarget{readme.md_autotoc_md23161}{}\doxysubsection{\texorpdfstring{When to use this}{When to use this}}\label{readme.md_autotoc_md23161}
This project is useful when you want to support autolink literals in markdown.

You can use these extensions when you are working with \href{https://github.com/micromark/micromark}{\texttt{ {\ttfamily micromark}}}. To support all GFM features, use \href{https://github.com/micromark/micromark-extension-gfm}{\texttt{ {\ttfamily micromark-\/extension-\/gfm}}} instead.

When you need a syntax tree, combine this package with \href{https://github.com/syntax-tree/mdast-util-gfm-autolink-literal}{\texttt{ {\ttfamily mdast-\/util-\/gfm-\/autolink-\/literal}}}.

All these packages are used in \href{https://github.com/remarkjs/remark-gfm}{\texttt{ {\ttfamily remark-\/gfm}}}, which focusses on making it easier to transform content by abstracting these internals away.\hypertarget{readme.md_autotoc_md23162}{}\doxysubsection{\texorpdfstring{Install}{Install}}\label{readme.md_autotoc_md23162}

GFM autolink literals relate to the {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} element in HTML. See \href{https://html.spec.whatwg.org/multipage/text-level-semantics.html\#the-a-element}{\texttt{ {\itshape § 4.\+5.\+1 The {\ttfamily a} element}}} in the HTML spec for more info. When an email autolink is used, the string {\ttfamily mailto\+:} is prepended when generating the {\ttfamily href} attribute of the hyperlink. When a www autolink is used, the string {\ttfamily \href{http://}{\texttt{ http\+://}}} is prepended.\hypertarget{readme.md_autotoc_md23172}{}\doxysubsection{\texorpdfstring{CSS}{CSS}}\label{readme.md_autotoc_md23172}

The grammar for GFM autolink literal is very relaxed\+: basically anything except for whitespace is allowed after a prefix. To use whitespace characters and otherwise impossible characters, in URLs, you can use percent encoding\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{https://example.com/alpha\%20bravo}

\end{DoxyCode}


Yields\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{<p><a\ href="{}https://example.com/alpha\%20bravo"{}>https://example.com/alpha\%20bravo</a></p>}

\end{DoxyCode}


There are several cases where incorrect encoding of URLs would, in other languages, result in a parse error. In markdown, there are no errors, and URLs are normalized. In addition, many characters are percent encoded (\href{https://github.com/micromark/micromark/tree/main/packages/micromark-util-sanitize-uri}{\texttt{ {\ttfamily sanitize\+Uri}}}). For example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{www.a👍b\%}

\end{DoxyCode}


Yields\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{<p><a\ href="{}http://www.a\%F0\%9F\%91\%8Db\%25"{}>www.a👍b\%</a></p>}

\end{DoxyCode}


There is a big difference between how www and protocol literals work compared to how email literals work. The first two are done when parsing, and work like anything else in markdown. But email literals are handled afterwards\+: when everything is parsed, we look back at the events to figure out if there were email addresses. This particularly affects how they interleave with character escapes and character references.\hypertarget{readme.md_autotoc_md23174}{}\doxysubsection{\texorpdfstring{Types}{Types}}\label{readme.md_autotoc_md23174}
