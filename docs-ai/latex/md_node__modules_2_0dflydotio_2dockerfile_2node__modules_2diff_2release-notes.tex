\item \href{https://github.com/kpdecker/jsdiff/pull/554}{\texttt{ \#554}} {\bfseries{{\ttfamily diff\+Words} treats numbers and underscores as word characters again.}} This behaviour was broken in v6.\+0.\+0.
\end{DoxyItemize}\hypertarget{md_node__modules_2_0dflydotio_2dockerfile_2node__modules_2diff_2release-notes_autotoc_md3341}{}\doxysection{\texorpdfstring{6.\+0.\+0}{6.\+0.\+0}}\label{md_node__modules_2_0dflydotio_2dockerfile_2node__modules_2diff_2release-notes_autotoc_md3341}
This is a release containing many, {\itshape many} breaking changes. The objective of this release was to carry out a mass fix, in one go, of all the open bugs and design problems that required breaking changes to fix. A substantial, but exhaustive, changelog is below.

\href{https://github.com/kpdecker/jsdiff/compare/v5.2.0...v6.0.0}{\texttt{ Commits}}


\begin{DoxyItemize}
\item \href{https://github.com/kpdecker/jsdiff/pull/497}{\texttt{ \#497}} {\bfseries{{\ttfamily diff\+Words} behavior has been radically changed.}} Previously, even with {\ttfamily ignore\+Whitespace\+: true}, runs of whitespace were tokens, which led to unhelpful and unintuitive diffing behavior in typical texts. Specifically, even when two texts contained overlapping passages, {\ttfamily diff\+Words} would sometimes choose to delete all the words from the old text and insert them anew in their new positions in order to avoid having to delete or insert whitespace tokens. Whitespace sequences are no longer tokens as of this release, which affects both the generated diffs and the {\ttfamily count}s.

Runs of whitespace are still tokens in {\ttfamily diff\+Words\+With\+Space}.

As part of the changes to {\ttfamily diff\+Words}, {\bfseries{a new {\ttfamily .post\+Process} method has been added on the base {\ttfamily Diff} type}}, which can be overridden in custom {\ttfamily Diff} implementations.

{\bfseries{{\ttfamily diff\+Lines} with {\ttfamily ignore\+Whitespace\+: true} will no longer ignore the insertion or deletion of entire extra lines of whitespace at the end of the text}}. Previously, these would not show up as insertions or deletions, as a side effect of a hack in the base diffing algorithm meant to help ignore whitespace in {\ttfamily diff\+Words}. More generally, {\bfseries{the undocumented special handling in the core algorithm for ignored terminals has been removed entirely.}} (This special case behavior used to rewrite the final two change objects in a scenario where the final change object was an addition or deletion and its {\ttfamily value} was treated as equal to the empty string when compared using the diff object\textquotesingle{}s {\ttfamily .equals} method.)
\item \href{https://github.com/kpdecker/jsdiff/pull/500}{\texttt{ \#500}} {\bfseries{{\ttfamily diff\+Chars} now diffs Unicode code points}} instead of UTF-\/16 code units.
\item \href{https://github.com/kpdecker/jsdiff/pull/508}{\texttt{ \#508}} {\bfseries{{\ttfamily parse\+Patch} now always runs in what was previously "{}strict"{} mode; the undocumented {\ttfamily strict} option has been removed.}} Previously, by default, {\ttfamily parse\+Patch} (and other patch functions that use it under the hood to parse patches) would accept a patch where the line counts in the headers were inconsistent with the actual patch content -\/ e.\+g. where a hunk started with the header {\ttfamily @@ -\/1,3 +1,6 @@}, indicating that the content below spanned 3 lines in the old file and 6 lines in the new file, but then the actual content below the header consisted of some different number of lines, say 10 lines of context, 5 deletions, and 1 insertion. Actually trying to work with these patches using {\ttfamily apply\+Patch} or {\ttfamily merge}, however, would produce incorrect results instead of just ignoring the incorrect headers, making this "{}feature"{} more of a trap than something actually useful. It\textquotesingle{}s been ripped out, and now we are always "{}strict"{} and will reject patches where the line counts in the headers aren\textquotesingle{}t consistent with the actual patch content.
\item \href{https://github.com/kpdecker/jsdiff/pull/435}{\texttt{ \#435}} {\bfseries{Fix {\ttfamily parse\+Patch} handling of control characters.}} {\ttfamily parse\+Patch} used to interpret various unusual control characters -\/ namely vertical tabs, form feeds, lone carriage returns without a line feed, and EBCDIC NELs -\/ as line breaks when parsing a patch file. This was inconsistent with the behavior of both Js\+Diff\textquotesingle{}s own {\ttfamily diff\+Lines} method and also the Unix {\ttfamily diff} and {\ttfamily patch} utils, which all simply treat those control characters as ordinary characters. The result of this discrepancy was that some well-\/formed patches -\/ produced either by {\ttfamily diff} or by Js\+Diff itself and handled properly by the {\ttfamily patch} util -\/ would be wrongly parsed by {\ttfamily parse\+Patch}, with the effect that it would disregard the remainder of a hunk after encountering one of these control characters.
\item \href{https://github.com/kpdecker/jsdiff/pull/439}{\texttt{ \#439}} {\bfseries{Prefer diffs that order deletions before insertions.}} When faced with a choice between two diffs with an equal total edit distance, the Myers diff algorithm generally prefers one that does deletions before insertions rather than insertions before deletions. For instance, when diffing {\ttfamily abcd} against {\ttfamily acbd}, it will prefer a diff that says to delete the {\ttfamily b} and then insert a new {\ttfamily b} after the {\ttfamily c}, over a diff that says to insert a {\ttfamily c} before the {\ttfamily b} and then delete the existing {\ttfamily c}. Js\+Diff deviated from the published Myers algorithm in a way that led to it having the opposite preference in many cases, including that example. This is now fixed, meaning diffs output by Js\+Diff will more accurately reflect what the published Myers diff algorithm would output.
\item \href{https://github.com/kpdecker/jsdiff/pull/455}{\texttt{ \#455}} {\bfseries{The {\ttfamily added} and {\ttfamily removed} properties of change objects are now guaranteed to be set to a boolean value.}} (Previously, they would be set to {\ttfamily undefined} or omitted entirely instead of setting them to false.)
\item \href{https://github.com/kpdecker/jsdiff/pull/464}{\texttt{ \#464}} Specifying {\ttfamily \{max\+Edit\+Length\+: 0\}} now sets a max edit length of 0 instead of no maximum.
\item \href{https://github.com/kpdecker/jsdiff/pull/460}{\texttt{ \#460}} {\bfseries{Added {\ttfamily one\+Change\+Per\+Token} option.}}
\item \href{https://github.com/kpdecker/jsdiff/pull/467}{\texttt{ \#467}} {\bfseries{Consistent ordering of arguments to {\ttfamily comparator(left, right)}.}} Values from the old array will now consistently be passed as the first argument ({\ttfamily left}) and values from the new array as the second argument ({\ttfamily right}). Previously this was almost (but not quite) always the other way round.
\item \href{https://github.com/kpdecker/jsdiff/pull/480}{\texttt{ \#480}} {\bfseries{Passing {\ttfamily max\+Edit\+Length} to {\ttfamily create\+Patch} \& {\ttfamily create\+Two\+Files\+Patch} now works properly}} (i.\+e. returns undefined if the max edit distance is exceeded; previous behavior was to crash with a {\ttfamily Type\+Error} if the edit distance was exceeded).
\item \href{https://github.com/kpdecker/jsdiff/pull/486}{\texttt{ \#486}} {\bfseries{The {\ttfamily ignore\+Whitespace} option of {\ttfamily diff\+Lines} behaves more sensibly now.}} {\ttfamily value}s in returned change objects now include leading/trailing whitespace even when {\ttfamily ignore\+Whitespace} is used, just like how with {\ttfamily ignore\+Case} the {\ttfamily value}s still reflect the case of one of the original texts instead of being all-\/lowercase. {\ttfamily ignore\+Whitespace} is also now compatible with {\ttfamily newline\+Is\+Token}. Finally, {\bfseries{{\ttfamily diff\+Trimmed\+Lines} is deprecated}} (and removed from the docs) in favour of using {\ttfamily diff\+Lines} with {\ttfamily ignore\+Whitespace\+: true}; the two are, and always have been, equivalent.
\item \href{https://github.com/kpdecker/jsdiff/pull/490}{\texttt{ \#490}} {\bfseries{When calling diffing functions in async mode by passing a {\ttfamily callback} option, the diff result will now be passed as the {\itshape first} argument to the callback instead of the second.}} (Previously, the first argument was never used at all and would always have value {\ttfamily undefined}.)
\item \href{github.com/kpdecker/jsdiff/pull/489}{\texttt{ \#489}} {\bfseries{{\ttfamily this.\+options} no longer exists on {\ttfamily Diff} objects.}} Instead, {\ttfamily options} is now passed as an argument to methods that rely on options, like {\ttfamily equals(left, right, options)}. This fixes a race condition in async mode, where diffing behaviour could be changed mid-\/execution if a concurrent usage of the same {\ttfamily Diff} instances overwrote its {\ttfamily options}.
\item \href{https://github.com/kpdecker/jsdiff/pull/518}{\texttt{ \#518}} {\bfseries{{\ttfamily linedelimiters} no longer exists}} on patch objects; instead, when a patch with Windows-\/style CRLF line endings is parsed, \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}the lines in {\ttfamily lines} will end with {\ttfamily \textbackslash{}r}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}. There is now a {\bfseries{new {\ttfamily auto\+Convert\+Line\+Endings} option, on by default}}, which makes it so that when a patch with Windows-\/style line endings is applied to a source file with Unix style line endings, the patch gets autoconverted to use Unix-\/style line endings, and when a patch with Unix-\/style line endings is applied to a source file with Windows-\/style line endings, it gets autoconverted to use Windows-\/style line endings.
\item \href{https://github.com/kpdecker/jsdiff/pull/521}{\texttt{ \#521}} {\bfseries{the {\ttfamily callback} option is now supported by {\ttfamily structured\+Patch}, {\ttfamily create\+Patch}}}
\item {\bfseries{{\ttfamily \mbox{[}\#529\mbox{]}(\href{https://github.com/kpdecker/jsdiff/pull/529}{\texttt{ https\+://github.\+com/kpdecker/jsdiff/pull/529}}) \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}parse\+Patch{\ttfamily can now parse patches where lines starting with}--{\ttfamily or}++\`{} are deleted/inserted}}; previously, there were edge cases where the parser would choke on valid patches or give wrong results.
\item \href{https://github.com/kpdecker/jsdiff/pull/530}{\texttt{ \#530}} {\bfseries{Added {\ttfamily ignore\+Newline\+At\+Eof} option{\ttfamily to}diff\+Lines\`{}}}
\item \href{https://github.com/kpdecker/jsdiff/pull/533}{\texttt{ \#533}} {\bfseries{{\ttfamily apply\+Patch} uses an entirely new algorithm for fuzzy matching.}} Differences between the old and new algorithm are as follows\+:
\begin{DoxyItemize}
\item The {\ttfamily fuzz\+Factor} now indicates the maximum \href{https://en.wikipedia.org/wiki/Levenshtein_distance}{\texttt{ {\itshape Levenshtein} distance}} that there can be between the context shown in a hunk and the actual file content at a location where we try to apply the hunk. (Previously, it represented a maximum \href{https://en.wikipedia.org/wiki/Hamming_distance}{\texttt{ {\itshape Hamming} distance}}, meaning that a single insertion or deletion in the source file could stop a hunk from applying even with a high {\ttfamily fuzz\+Factor}.)
\item A hunk containing a deletion can now only be applied in a context where the line to be deleted actually appears verbatim. (Previously, as long as enough context lines in the hunk matched, {\ttfamily apply\+Patch} would apply the hunk anyway and delete a completely different line.)
\item The context line immediately before and immediately after an insertion must match exactly between the hunk and the file for a hunk to apply. (Previously this was not required.)
\end{DoxyItemize}
\item \href{https://github.com/kpdecker/jsdiff/pull/535}{\texttt{ \#535}} {\bfseries{A bug in patch generation functions is now fixed}} that would sometimes previously cause {\ttfamily \textbackslash{} No newline at end of file} to appear in the wrong place in the generated patch, resulting in the patch being invalid.
\item \href{https://github.com/kpdecker/jsdiff/pull/535}{\texttt{ \#535}} {\bfseries{Passing {\ttfamily newline\+Is\+Token\+: true} to {\itshape patch}-\/generation functions is no longer allowed.}} (Passing it to {\ttfamily diff\+Lines} is still supported -\/ it\textquotesingle{}s only functions like {\ttfamily create\+Patch} where passing {\ttfamily newline\+Is\+Token} is now an error.) Allowing it to be passed never really made sense, since in cases where the option had any effect on the output at all, the effect tended to be causing a garbled patch to be created that couldn\textquotesingle{}t actually be applied to the source file.
\item \href{https://github.com/kpdecker/jsdiff/pull/539}{\texttt{ \#539}} {\bfseries{{\ttfamily diff\+Words} now takes an optional {\ttfamily intl\+Segmenter} option}} which should be an {\ttfamily Intl.\+Segmenter} with word-\/level granularity. This provides better tokenization of text into words than the default behaviour, even for English but especially for some other languages for which the default behaviour is poor.
\end{DoxyItemize}\hypertarget{md_node__modules_2_0dflydotio_2dockerfile_2node__modules_2diff_2release-notes_autotoc_md3342}{}\doxysection{\texorpdfstring{v5.\+2.\+0}{v5.\+2.\+0}}\label{md_node__modules_2_0dflydotio_2dockerfile_2node__modules_2diff_2release-notes_autotoc_md3342}

\begin{DoxyItemize}
\item \href{https://github.com/kpdecker/jsdiff/pull/411}{\texttt{ \#411}} Big performance improvement. Previously an O(n) array-\/copying operation inside the innermost loop of jsdiff\textquotesingle{}s base diffing code increased the overall worst-\/case time complexity of computing a diff from O(n²) to O(n³). This is now fixed, bringing the worst-\/case time complexity down to what it theoretically should be for a Myers diff implementation.
\item \href{https://github.com/kpdecker/jsdiff/pull/411}{\texttt{ \#448}} Performance improvement. Diagonals whose furthest-\/reaching D-\/path would go off the edge of the edit graph are now skipped, rather than being pointlessly considered as called for by the original Myers diff algorithm. This dramatically speeds up computing diffs where the new text just appends or truncates content at the end of the old text.
\item \href{https://github.com/kpdecker/jsdiff/issues/351}{\texttt{ \#351}} Importing from the lib folder -\/ e.\+g. {\ttfamily require("{}diff/lib/diff/word.\+js"{})} -\/ will work again now. This had been broken for users on the latest version of Node since Node 17.\+5.\+0, which changed how Node interprets the {\ttfamily exports} property in jsdiff\textquotesingle{}s {\ttfamily package.\+json} file.
\item \href{https://github.com/kpdecker/jsdiff/issues/344}{\texttt{ \#344}} {\ttfamily diff\+Lines}, {\ttfamily create\+Two\+Files\+Patch}, and other patch-\/creation methods now take an optional {\ttfamily strip\+Trailing\+Cr\+: true} option which causes Windows-\/style {\ttfamily \textbackslash{}r\textbackslash{}n} line endings to be replaced with Unix-\/style {\ttfamily \textbackslash{}n} line endings before calculating the diff, just like GNU {\ttfamily diff}\textquotesingle{}s {\ttfamily -\/-\/strip-\/trailing-\/cr} flag.
