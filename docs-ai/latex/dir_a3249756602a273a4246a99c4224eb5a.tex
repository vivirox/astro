This package contains extensions that add support for the table syntax enabled by GFM to \href{https://github.com/micromark/micromark}{\texttt{ {\ttfamily micromark}}}. These extensions match github.\+com.\hypertarget{readme.md_autotoc_md23241}{}\doxysubsection{\texorpdfstring{When to use this}{When to use this}}\label{readme.md_autotoc_md23241}
This project is useful when you want to support tables in markdown.

You can use these extensions when you are working with \href{https://github.com/micromark/micromark}{\texttt{ {\ttfamily micromark}}}. To support all GFM features, use \href{https://github.com/micromark/micromark-extension-gfm}{\texttt{ {\ttfamily micromark-\/extension-\/gfm}}} instead.

When you need a syntax tree, combine this package with \href{https://github.com/syntax-tree/mdast-util-gfm-table}{\texttt{ {\ttfamily mdast-\/util-\/gfm-\/table}}}.

All these packages are used in \href{https://github.com/remarkjs/remark-gfm}{\texttt{ {\ttfamily remark-\/gfm}}}, which focusses on making it easier to transform content by abstracting these internals away.\hypertarget{readme.md_autotoc_md23242}{}\doxysubsection{\texorpdfstring{Install}{Install}}\label{readme.md_autotoc_md23242}

\end{DoxyItemize}\hypertarget{readme.md_autotoc_md23250}{}\doxysubsection{\texorpdfstring{Authoring}{Authoring}}\label{readme.md_autotoc_md23250}
When authoring markdown with GFM tables, it’s recommended to {\itshape always} put pipes around cells. Without them, it can be hard to infer whether the table will work, how many columns there are, and which column you are currently editing.

It is recommended to not use many columns, as it results in very long lines, making it hard to infer which column you are currently editing.

For larger tables, particularly when cells vary in size, it is recommended {\itshape not} to manually “pad” cell text. While it can look better, it results in a lot of time spent realigning everything when a new, longer cell is added or the longest cell removed, as every row then must be changed. Other than costing time, it also causes large diffs in Git.

To illustrate, when authoring large tables, it is discouraged to pad cells like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{|\ Alpha\ bravo\ charlie\ |\ \ \ \ \ \ \ \ \ \ \ \ \ \ delta\ |}
\DoxyCodeLine{|\ -\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/\ |\ -\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/:\ |}
\DoxyCodeLine{|\ Echo\ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ |\ Foxtrot\ golf\ hotel\ |}

\end{DoxyCode}


Instead, use single spaces (and single filler dashes)\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{|\ Alpha\ bravo\ charlie\ |\ delta\ |}
\DoxyCodeLine{|\ -\/\ |\ -\/:\ |}
\DoxyCodeLine{|\ Echo\ |\ Foxtrot\ golf\ hotel\ |}

\end{DoxyCode}

\hypertarget{readme.md_autotoc_md23253}{}\doxysubsection{\texorpdfstring{Syntax}{Syntax}}\label{readme.md_autotoc_md23253}
Tables form with the following BNF\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{gfmTable\ ::=\ gfmTableHead\ 0*(eol\ gfmTableBodyRow)}
\DoxyCodeLine{}
\DoxyCodeLine{;\ Restriction:\ both\ rows\ must\ have\ the\ same\ number\ of\ cells.}
\DoxyCodeLine{gfmTableHead\ ::=\ gfmTableRow\ eol\ gfmTableDelimiterRow}
\DoxyCodeLine{}
\DoxyCodeLine{gfmTableRow\ ::=\ ["{}|"{}]\ gfmTableCell\ 0*("{}|"{}\ gfmTableCell)\ ["{}|"{}]\ *spaceOrTab}
\DoxyCodeLine{gfmTableCell\ ::=\ *spaceOrTab\ gfmTableText\ *spaceOrTab}
\DoxyCodeLine{gfmTableText\ ::=\ 0*(line\ -\/\ "{}\(\backslash\)\(\backslash\)"{}\ -\/\ "{}|"{}\ /\ "{}\(\backslash\)\(\backslash\)"{}\ ["{}\(\backslash\)\(\backslash\)"{}\ /\ "{}|"{}])}
\DoxyCodeLine{}
\DoxyCodeLine{gfmTableDelimiterRow\ ::=\ ["{}|"{}]\ gfmTableDelimiterCell\ 0*("{}|"{}\ gfmTableDelimiterCell)\ ["{}|"{}]\ *spaceOrTab}
\DoxyCodeLine{gfmTableDelimiterCell\ ::=\ *spaceOrTab\ gfmTableDelimiterValue\ *spaceOrTab}
\DoxyCodeLine{gfmTableDelimiterValue\ ::=\ ["{}:"{}]\ 1*"{}-\/"{}\ ["{}:"{}]}

\end{DoxyCode}


As this construct occurs in flow, like all flow constructs, it must be followed by an eol (line ending) or eof (end of file).

The above grammar shows that basically anything can be a cell or a row. The main thing that makes something a row, is that it occurs directly before or after a delimiter row, or after another row.

It is not required for a table to have a body\+: it can end right after the delimiter row.

Each column can be marked with an alignment. The alignment marker is a colon ({\ttfamily \+:}) used before and/or after delimiter row filler. To illustrate\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{|\ none\ |\ left\ |\ right\ |\ center\ |}
\DoxyCodeLine{|\ -\/-\/-\/-\/\ |\ :-\/-\/-\/\ |\ -\/-\/-\/-\/:\ |\ :-\/-\/-\/-\/:\ |}

\end{DoxyCode}


The number of cells in the delimiter row, is the number of columns of the table. Only the head row is required to have the same number of cells. Body rows are not required to have a certain number of cells. For body rows that have less cells than the number of columns of the table, empty cells are injected. When a row has more cells than the number of columns of the table, the superfluous cells are dropped. To illustrate\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{|\ a\ |\ b\ |}
\DoxyCodeLine{|\ -\/\ |\ -\/\ |}
\DoxyCodeLine{|\ c\ |}
\DoxyCodeLine{|\ d\ |\ e\ |\ f\ |}

\end{DoxyCode}

Each cell’s text is interpreted as the \href{https://github.com/micromark/micromark\#content-types}{\texttt{ text}} content type. That means that it can include constructs such as attention (emphasis, strong).

The grammar for cells prohibits the use of {\ttfamily \texorpdfstring{$\vert$}{|}} in them. To use pipes in cells, encode them as a character reference or character escape\+: {\ttfamily \&vert;} (or {\ttfamily \&Vertical\+Line;}, {\ttfamily \&verbar;}, {\ttfamily \&\#124;}, {\ttfamily \&\#x7c;}) or {\ttfamily \textbackslash{}\texorpdfstring{$\vert$}{|}}.

Escapes will typically work, but they are not supported in code (text) (and the math (text) extension). To work around this, Git\+Hub came up with a rather weird “trick”. When inside a table cell {\itshape and} inside code, escaped pipes {\itshape are} decoded. To illustrate\+:
