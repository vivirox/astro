⚠️ Middleware is unstable and should not be adopted in production. There is at least one known de-\/optimization in route module loading for {\ttfamily client\+Middleware} that we will be addressing this before a stable release.

⚠️ Enabling middleware contains a breaking change to the {\ttfamily context} parameter passed to your {\ttfamily loader}/{\ttfamily action} functions -\/ see below for more information.

Once enabled, routes can define an array of middleware functions that will run sequentially before route handlers run. These functions accept the same parameters as {\ttfamily loader}/{\ttfamily action} plus an additional {\ttfamily next} parameter to run the remaining data pipeline. This allows middlewares to perform logic before and after handlers execute.

\DoxyCodeLine{//\ Framework\ mode}
\DoxyCodeLine{export\ const\ unstable\_middleware\ =\ [serverLogger,\ serverAuth];\ //\ server}
\DoxyCodeLine{export\ const\ unstable\_clientMiddleware\ =\ [clientLogger];\ //\ client}
\DoxyCodeLine{}
\DoxyCodeLine{//\ Library\ mode}
\DoxyCodeLine{const\ routes\ =\ [}
\DoxyCodeLine{\ \ \{}
\DoxyCodeLine{\ \ \ \ path:\ "{}/"{},}
\DoxyCodeLine{\ \ \ \ //\ Middlewares\ are\ client-\/side\ for\ library\ mode\ SPA's}

\DoxyCodeLine{\ \ \ \ loader:\ rootLoader,}
\DoxyCodeLine{\ \ \ \ Component:\ Root,}

\DoxyCodeLine{\ \ \{\ request\ \},}

\DoxyCodeLine{}

\DoxyCodeLine{\};}

\end{DoxyCode}


Note that in the above example, the {\ttfamily next}/{\ttfamily middleware} functions don\textquotesingle{}t return anything. This is by design as on the client there is no "{}response"{} to send over the network like there would be for middlewares running on the server. The data is all handled behind the scenes by the stateful {\ttfamily router}.

For a server-\/side middleware, the {\ttfamily next} function will return the HTTP {\ttfamily Response} that React Router will be sending across the wire, thus giving you a chance to make changes as needed. You may throw a new response to short circuit and respond immediately, or you may return a new or altered response to override the default returned by {\ttfamily next()}.

\DoxyCodeLine{}

\DoxyCodeLine{\ \ let\ res\ =\ await\ next();}
\DoxyCodeLine{}

\DoxyCodeLine{}

\item Fix typegen for repeated params (\href{https://github.com/remix-run/react-router/pull/13012}{\texttt{ \#13012}})

In React Router, path parameters are keyed by their name. So for a path pattern like {\ttfamily /a/\+:id/b/\+:id?/c/\+:id}, the last {\ttfamily \+:id} will set the value for {\ttfamily id} in {\ttfamily use\+Params} and the {\ttfamily params} prop. For example, {\ttfamily /a/1/b/2/c/3} will result in the value {\ttfamily \{ id\+: 3 \}} at runtime.

Previously, generated types for params incorrectly modeled repeated params with an array. So {\ttfamily /a/1/b/2/c/3} generated a type like {\ttfamily \{ id\+: \mbox{[}1,2,3\mbox{]} \}}.

To be consistent with runtime behavior, the generated types now correctly model the "{}last one wins"{} semantics of path parameters. So {\ttfamily /a/1/b/2/c/3} now generates a type like {\ttfamily \{ id\+: 3 \}}.

React Router now generates types for each of your route modules. You can access those types by importing them from {\ttfamily ./+types.\texorpdfstring{$<$}{<}route filename without extension\texorpdfstring{$>$}{>}}. For example\+:

\end{DoxyCode}


This initial implementation targets type inference for\+:
\begin{DoxyItemize}
\item {\ttfamily Params} \+: Path parameters from your routing config in {\ttfamily routes.\+ts} including file-\/based routing
\item {\ttfamily Loader\+Data} \+: Loader data from {\ttfamily loader} and/or {\ttfamily client\+Loader} within your route module
\item {\ttfamily Action\+Data} \+: Action data from {\ttfamily action} and/or {\ttfamily client\+Action} within your route module
\end{DoxyItemize}

In the future, we plan to add types for the rest of the route module exports\+: {\ttfamily meta}, {\ttfamily links}, {\ttfamily headers}, {\ttfamily should\+Revalidate}, etc. We also plan to generate types for typesafe {\ttfamily Link}s\+:

\begin{DoxyItemize}
\item Add a new {\ttfamily future.\+v7\+\_\+relative\+Splat\+Path} flag to implement a breaking bug fix to relative routing when inside a splat route. (\href{https://github.com/remix-run/react-router/pull/11087}{\texttt{ \#11087}})

This fix was originally added in \href{https://github.com/remix-run/react-router/issues/10983}{\texttt{ \#10983}} and was later reverted in \href{https://github.com/remix-run/react-router/pull/11078}{\texttt{ \#11078}} because it was determined that a large number of existing applications were relying on the buggy behavior (see \href{https://github.com/remix-run/react-router/issues/11052}{\texttt{ \#11052}})

{\bfseries{The Bug}} The buggy behavior is that without this flag, the default behavior when resolving relative paths is to {\itshape ignore} any splat ({\ttfamily \texorpdfstring{$\ast$}{*}}) portion of the current route path.

{\bfseries{The Background}} This decision was originally made thinking that it would make the concept of nested different sections of your apps in {\ttfamily \texorpdfstring{$<$}{<}Routes\texorpdfstring{$>$}{>}} easier if relative routing would {\itshape replace} the current splat\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{<BrowserRouter>}
\DoxyCodeLine{\ \ <Routes>}
\DoxyCodeLine{\ \ \ \ <Route\ path="{}/"{}\ element=\{<Home\ />\}\ />}
\DoxyCodeLine{\ \ \ \ <Route\ path="{}dashboard/*"{}\ element=\{<Dashboard\ />\}\ />}
\DoxyCodeLine{\ \ </Routes>}
\DoxyCodeLine{</BrowserRouter>}

\end{DoxyCode}


Any paths like {\ttfamily /dashboard}, {\ttfamily /dashboard/team}, {\ttfamily /dashboard/projects} will match the {\ttfamily Dashboard} route. The dashboard component itself can then render nested {\ttfamily \texorpdfstring{$<$}{<}Routes\texorpdfstring{$>$}{>}}\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{function\ Dashboard()\ \{}
\DoxyCodeLine{\ \ return\ (}
\DoxyCodeLine{\ \ \ \ <div>}
\DoxyCodeLine{\ \ \ \ \ \ <h2>Dashboard</h2>}
\DoxyCodeLine{\ \ \ \ \ \ <nav>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Link\ to="{}/"{}>Dashboard\ Home</Link>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Link\ to="{}team"{}>Team</Link>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Link\ to="{}projects"{}>Projects</Link>}
\DoxyCodeLine{\ \ \ \ \ \ </nav>}
\DoxyCodeLine{}
\DoxyCodeLine{\ \ \ \ \ \ <Routes>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Route\ path="{}/"{}\ element=\{<DashboardHome\ />\}\ />}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Route\ path="{}team"{}\ element=\{<DashboardTeam\ />\}\ />}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Route\ path="{}projects"{}\ element=\{<DashboardProjects\ />\}\ />}
\DoxyCodeLine{\ \ \ \ \ \ </Routes>}
\DoxyCodeLine{\ \ \ \ </div>}
\DoxyCodeLine{\ \ );}
\DoxyCodeLine{\}}

\end{DoxyCode}


Now, all links and route paths are relative to the router above them. This makes code splitting and compartmentalizing your app really easy. You could render the {\ttfamily Dashboard} as its own independent app, or embed it into your large app without making any changes to it.

{\bfseries{The Problem}}

The problem is that this concept of ignoring part of a path breaks a lot of other assumptions in React Router -\/ namely that {\ttfamily "{}."{}} always means the current location pathname for that route. When we ignore the splat portion, we start getting invalid paths when using {\ttfamily "{}."{}}\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{//\ If\ we\ are\ on\ URL\ /dashboard/team,\ and\ we\ want\ to\ link\ to\ /dashboard/team:}
\DoxyCodeLine{function\ DashboardTeam()\ \{}
\DoxyCodeLine{\ \ //\ ❌\ This\ is\ broken\ and\ results\ in\ <a\ href="{}/dashboard"{}>}
\DoxyCodeLine{\ \ return\ <Link\ to="{}."{}>A\ broken\ link\ to\ the\ Current\ URL</Link>;}
\DoxyCodeLine{}
\DoxyCodeLine{\ \ //\ ✅\ This\ is\ fixed\ but\ super\ unintuitive\ since\ we're\ already\ at\ /dashboard/team!}
\DoxyCodeLine{\ \ return\ <Link\ to="{}./team"{}>A\ broken\ link\ to\ the\ Current\ URL</Link>;}
\DoxyCodeLine{\}}

\end{DoxyCode}


We\textquotesingle{}ve also introduced an issue that we can no longer move our {\ttfamily Dashboard\+Team} component around our route hierarchy easily -\/ since it behaves differently if we\textquotesingle{}re underneath a non-\/splat route, such as {\ttfamily /dashboard/\+:widget}. Now, our {\ttfamily "{}."{}} links will, properly point to ourself {\itshape inclusive of the dynamic param value} so behavior will break from it\textquotesingle{}s corresponding usage in a {\ttfamily /dashboard/\texorpdfstring{$\ast$}{*}} route.

Even worse, consider a nested splat route configuration\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{<BrowserRouter>}
\DoxyCodeLine{\ \ <Routes>}
\DoxyCodeLine{\ \ \ \ <Route\ path="{}dashboard"{}>}
\DoxyCodeLine{\ \ \ \ \ \ <Route\ path="{}*"{}\ element=\{<Dashboard\ />\}\ />}
\DoxyCodeLine{\ \ \ \ </Route>}
\DoxyCodeLine{\ \ </Routes>}
\DoxyCodeLine{</BrowserRouter>}

\end{DoxyCode}


Now, a {\ttfamily \texorpdfstring{$<$}{<}Link to="{}."{}\texorpdfstring{$>$}{>}} and a {\ttfamily \texorpdfstring{$<$}{<}Link to="{}.."{}\texorpdfstring{$>$}{>}} inside the {\ttfamily Dashboard} component go to the same place! That is definitely not correct!

Another common issue arose in Data Routers (and Remix) where any {\ttfamily \texorpdfstring{$<$}{<}Form\texorpdfstring{$>$}{>}} should post to it\textquotesingle{}s own route {\ttfamily action} if you the user doesn\textquotesingle{}t specify a form action\+:


\begin{DoxyCode}{0}

\DoxyCodeLine{\ \ \ \ \ \ \},}
\DoxyCodeLine{\ \ \ \ \},}

This is just a compounded issue from the above because the default location for a {\ttfamily Form} to submit to is itself ({\ttfamily "{}."{}}) -\/ and if we ignore the splat portion, that now resolves to the parent route.

{\bfseries{The Solution}} If you are leveraging this behavior, it\textquotesingle{}s recommended to enable the future flag, move your splat to it\textquotesingle{}s own route, and leverage {\ttfamily ../} for any links to "{}sibling"{} pages\+:

\DoxyCodeLine{\ \ \ \ <Route\ path="{}dashboard"{}>}
\DoxyCodeLine{\ \ \ \ \ \ <Route\ index\ path="{}*"{}\ element=\{<Dashboard\ />\}\ />}
\DoxyCodeLine{\ \ \ \ </Route>}
\DoxyCodeLine{\ \ </Routes>}
\DoxyCodeLine{</BrowserRouter>}
\DoxyCodeLine{}
\DoxyCodeLine{function\ Dashboard()\ \{}
\DoxyCodeLine{\ \ return\ (}
\DoxyCodeLine{\ \ \ \ <div>}
\DoxyCodeLine{\ \ \ \ \ \ <h2>Dashboard</h2>}
\DoxyCodeLine{\ \ \ \ \ \ <nav>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Link\ to="{}.."{}>Dashboard\ Home</Link>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Link\ to="{}../team"{}>Team</Link>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Link\ to="{}../projects"{}>Projects</Link>}
\DoxyCodeLine{\ \ \ \ \ \ </nav>}
\DoxyCodeLine{}
\DoxyCodeLine{\ \ \ \ \ \ <Routes>}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Route\ path="{}/"{}\ element=\{<DashboardHome\ />\}\ />}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Route\ path="{}team"{}\ element=\{<DashboardTeam\ />\}\ />}
\DoxyCodeLine{\ \ \ \ \ \ \ \ <Route\ path="{}projects"{}\ element=\{<DashboardProjects\ />\}\ />}
\DoxyCodeLine{\ \ \ \ \ \ </Router>}
\DoxyCodeLine{\ \ \ \ </div>}
\DoxyCodeLine{\ \ );}
\DoxyCodeLine{\}}

\end{DoxyCode}


This way, {\ttfamily .} means "{}the full current pathname for my route"{} in all cases (including static, dynamic, and splat routes) and {\ttfamily ..} always means "{}my parents pathname"{}.

\item {\bfseries{Introducing Lazy Route Modules!}} (\href{https://github.com/remix-run/react-router/pull/10045}{\texttt{ \#10045}})

In order to keep your application bundles small and support code-\/splitting of your routes, we\textquotesingle{}ve introduced a new {\ttfamily lazy()} route property. This is an async function that resolves the non-\/route-\/matching portions of your route definition ({\ttfamily loader}, {\ttfamily action}, {\ttfamily element}/{\ttfamily Component}, {\ttfamily error\+Element}/{\ttfamily Error\+Boundary}, {\ttfamily should\+Revalidate}, {\ttfamily handle}).

Lazy routes are resolved on initial load and during the {\ttfamily loading} or {\ttfamily submitting} phase of a navigation or fetcher call. You cannot lazily define route-\/matching properties ({\ttfamily path}, {\ttfamily index}, {\ttfamily children}) since we only execute your lazy route functions after we\textquotesingle{}ve matched known routes.

Your {\ttfamily lazy} functions will typically return the result of a dynamic import.

\DoxyCodeLine{//\ in\ our\ critical-\/path\ bundle,\ but\ then\ we\ lazily\ load\ modules\ for\ /a\ and\ /b\ so}
\DoxyCodeLine{//\ they\ don't\ load\ until\ the\ user\ navigates\ to\ those\ routes}
