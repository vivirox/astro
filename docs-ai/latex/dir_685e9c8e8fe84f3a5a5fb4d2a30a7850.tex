This package is a \href{http://reactjs.org}{\texttt{ React}} component that can be given a string of markdown that it’ll safely render to React elements. You can pass plugins to change how markdown is transformed and pass components that will be used instead of normal HTML elements.


\begin{DoxyItemize}
\item to learn markdown, see this \href{https://commonmark.org/help/}{\texttt{ cheatsheet and tutorial}}
\item to try out {\ttfamily react-\/markdown}, see \href{https://remarkjs.github.io/react-markdown/}{\texttt{ our demo}}
\end{DoxyItemize}\hypertarget{readme.md_autotoc_md26880}{}\doxysubsubsection{\texorpdfstring{When should I use this?}{When should I use this?}}\label{readme.md_autotoc_md26880}
There are other ways to use markdown in React out there so why use this one? The three main reasons are that they often rely on {\ttfamily dangerously\+Set\+Inner\+HTML}, have bugs with how they handle markdown, or don’t let you swap elements for components. {\ttfamily react-\/markdown} builds a virtual DOM, so React only replaces what changed, from a syntax tree. That’s supported because we use \href{https://github.com/unifiedjs/unified}{\texttt{ unified}}, specifically \href{https://github.com/remarkjs/remark}{\texttt{ remark}} for markdown and \href{https://github.com/rehypejs/rehype}{\texttt{ rehype}} for HTML, which are popular tools to transform content with plugins.

This package focusses on making it easy for beginners to safely use markdown in React. When you’re familiar with unified, you can use a modern hooks based alternative \href{https://github.com/remarkjs/react-remark}{\texttt{ {\ttfamily react-\/remark}}} or \href{https://github.com/rehypejs/rehype-react}{\texttt{ {\ttfamily rehype-\/react}}} manually. If you instead want to use Java\+Script and JSX {\itshape inside} markdown files, use \href{https://github.com/mdx-js/mdx/}{\texttt{ MDX}}.\hypertarget{readme.md_autotoc_md26881}{}\doxysubsubsection{\texorpdfstring{Install}{Install}}\label{readme.md_autotoc_md26881}

{\ttfamily react-\/markdown} is a unified pipeline — wrapped so that most folks don’t need to directly interact with unified. The processor goes through these steps\+:

\item parse markdown to mdast (markdown syntax tree)
\item transform through remark (markdown ecosystem)
\item transform mdast to hast (HTML syntax tree)
\item transform through rehype (HTML ecosystem)
\item render hast to React with components

The keys in components are HTML equivalents for the things you write with markdown (such as {\ttfamily h1} for {\ttfamily \# heading}). Normally, in markdown, those are\+: {\ttfamily a}, {\ttfamily blockquote}, {\ttfamily br}, {\ttfamily code}, {\ttfamily em}, {\ttfamily h1}, {\ttfamily h2}, {\ttfamily h3}, {\ttfamily h4}, {\ttfamily h5}, {\ttfamily h6}, {\ttfamily hr}, {\ttfamily img}, {\ttfamily li}, {\ttfamily ol}, {\ttfamily p}, {\ttfamily pre}, {\ttfamily strong}, and {\ttfamily ul}. Other remark or rehype plugins that add support for new constructs will also work with {\ttfamily react-\/markdown}.

The props that are passed are what you probably would expect\+: an {\ttfamily a} (link) will get {\ttfamily href} (and {\ttfamily title}) props, and {\ttfamily img} (image) an {\ttfamily src}, {\ttfamily alt} and {\ttfamily title}, etc.
