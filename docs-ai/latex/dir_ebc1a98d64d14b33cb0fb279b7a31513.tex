unified is two things\+:


\begin{DoxyItemize}
\item {\bfseries{unified}} is a collective of 500+ free and open source packages that work with content as structured data (ASTs)
\item {\ttfamily unified} (this project) is the core package, used in 1.\+3m+ projects on GH, to process content with plugins
\end{DoxyItemize}

Several ecosystems are built on unified around different kinds of content. Notably, \href{https://github.com/remarkjs/remark}{\texttt{ remark}} (markdown), \href{https://github.com/rehypejs/rehype}{\texttt{ rehype}} (HTML), and \href{https://github.com/retextjs/retext}{\texttt{ retext}} (natural language). These ecosystems can be connected together.


\begin{DoxyItemize}
\item for more about us, see \href{https://unifiedjs.com}{\texttt{ {\ttfamily unifiedjs.\+com}}}
\item for updates, see \href{https://twitter.com/unifiedjs}{\texttt{ @unifiedjs}} on Twitter
\item for questions, see \href{https://github.com/unifiedjs/.github/blob/main/support.md}{\texttt{ support}}
\item to help, see \doxylink{readme.md_contribute}{contribute} and sponsor below

In some cases, you are already using unified. For example, it‚Äôs used in MDX, Gatsby, Docusaurus, etc. In those cases, you don‚Äôt need to add {\ttfamily unified} yourself but you can include plugins into those projects.

\hypertarget{readme.md_autotoc_md34149}{}\doxysubsection{\texorpdfstring{Overview}{Overview}}\label{readme.md_autotoc_md34149}
{\ttfamily unified} is an interface for processing content with syntax trees. Syntax trees are a representation of content understandable to programs. Those programs, called {\itshape plugins}, take these trees and inspect and modify them. To get to the syntax tree from text, there is a {\itshape parser}. To get from that back to text, there is a {\itshape compiler}. This is the {\itshape process} of a {\itshape processor}.

Processors process content. On its own, {\ttfamily unified} (the root processor) doesn‚Äôt work. It needs to be configured with plugins to work. For example\+:

That processor can do different things. It can\+:


\begin{DoxyItemize}
\item ‚Ä¶parse markdown ({\ttfamily parse})
\item ‚Ä¶turn parsed markdown into HTML and format the HTML ({\ttfamily run})
\item ‚Ä¶compile HTML ({\ttfamily stringify})
\item ‚Ä¶do all of the above ({\ttfamily process})
\end{DoxyItemize}

Every processor implements another processor. To create a processor, call another processor. The new processor is configured to work the same as its ancestor. But when the descendant processor is configured in the future it does not affect the ancestral processor.

When processors are exposed from a module (for example, {\ttfamily unified} itself) they should not be configured directly, as that would change their behavior for all module users. Those processors are {\itshape frozen} and they should be called to create a new processor before they are used.\hypertarget{readme.md_autotoc_md34151}{}\doxysubsubsubsubsubsection{\texorpdfstring{File}{File}}\label{readme.md_autotoc_md34151}
When processing a document, metadata is gathered about that document. \href{https://github.com/vfile/vfile}{\texttt{ {\ttfamily vfile}}} is the file format that stores data, metadata, and messages about files for unified and plugins.

There are several \href{https://github.com/vfile/vfile\#list-of-utilities}{\texttt{ utilities}} for working with these files.\hypertarget{readme.md_autotoc_md34152}{}\doxysubsubsubsubsubsection{\texorpdfstring{Syntax tree}{Syntax tree}}\label{readme.md_autotoc_md34152}
The syntax trees used in unified are \href{https://github.com/syntax-tree/unist}{\texttt{ unist}} nodes. A tree represents a whole document and each \href{https://github.com/syntax-tree/unist\#node}{\texttt{ node}} is a plain Java\+Script object with a {\ttfamily type} field. The semantics of nodes and the format of syntax trees is defined by other projects\+:


\begin{DoxyItemize}
\item \href{https://github.com/syntax-tree/esast}{\texttt{ esast}} ‚Äî Java\+Script
\item \href{https://github.com/syntax-tree/hast}{\texttt{ hast}} ‚Äî HTML
\item \href{https://github.com/syntax-tree/mdast}{\texttt{ mdast}} ‚Äî markdown
\item \href{https://github.com/syntax-tree/nlcst}{\texttt{ nlcst}} ‚Äî natural language
\item \href{https://github.com/syntax-tree/xast}{\texttt{ xast}} ‚Äî XML
\end{DoxyItemize}

There are many utilities for working with trees listed in each aforementioned project and maintained in the \href{https://github.com/syntax-tree}{\texttt{ {\ttfamily syntax-\/tree}}} organization. These utilities are a level lower than unified itself and are building blocks that can be used to make plugins.

\hypertarget{readme.md_autotoc_md34158}{}\doxysubsubsubsubsubsection{\texorpdfstring{Transforming between ecosystems}{Transforming between ecosystems}}\label{readme.md_autotoc_md34158}
Ecosystems can be combined in two modes.

{\bfseries{Bridge}} mode transforms the tree from one format ({\itshape origin}) to another ({\itshape destination}). A different processor runs on the destination tree. Afterwards, the original processor continues with the origin tree.

{\bfseries{Mutate}} mode also transforms the syntax tree from one format to another. But the original processor continues transforming the destination tree.

In the previous example (‚Äú\+Programming interface‚Äù), {\ttfamily remark-\/retext} is used in bridge mode\+: the origin syntax tree is kept after retext is done; whereas {\ttfamily remark-\/rehype} is used in mutate mode\+: it sets a new syntax tree and discards the origin tree.

The following plugins lets you combine ecosystems\+:


\begin{DoxyItemize}
\item \href{https://github.com/remarkjs/remark-retext}{\texttt{ {\ttfamily remark-\/retext}}} ‚Äî turn markdown into natural language
\item \href{https://github.com/remarkjs/remark-rehype}{\texttt{ {\ttfamily remark-\/rehype}}} ‚Äî turn markdown into HTML
\item \href{https://github.com/rehypejs/rehype-retext}{\texttt{ {\ttfamily rehype-\/retext}}} ‚Äî turn HTML into natural language
\item \href{https://github.com/rehypejs/rehype-remark}{\texttt{ {\ttfamily rehype-\/remark}}} ‚Äî turn HTML into markdown

Transformers handle syntax trees and files (Type\+Script type).

They are functions that are called each time a syntax tree and file are passed through the run phase. When an error occurs in them (either because it‚Äôs thrown, returned, rejected, or passed to {\ttfamily next}), the process stops.

The run phase is handled by \href{https://github.com/wooorm/trough\#function-fninput-next}{\texttt{ {\ttfamily trough}}}, see its documentation for the exact semantics of these functions.
