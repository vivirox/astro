The main focus of this version. Addresses the most demanded user requests (\href{https://github.com/html-to-text/node-html-to-text/issues/159}{\texttt{ \#159}}, \href{https://github.com/html-to-text/node-html-to-text/issues/179}{\texttt{ \#179}}, partially \href{https://github.com/html-to-text/node-html-to-text/issues/143}{\texttt{ \#143}}).

It is now possible to specify formatting options or assign custom formatters not only by tag names but by almost any selectors.

See the README \href{https://github.com/html-to-text/node-html-to-text\#selectors}{\texttt{ Selectors}} section for details.

Note\+: The new {\ttfamily selectors} option is an array, in contrast to the {\ttfamily tags} option introduced in version 6 (and now deprecated). Selectors have to have a well defined order and object properties is not a right tool for that.

Two new packages were created to enable this feature -\/ \href{https://github.com/mxxii/parseley}{\texttt{ parseley}} and \href{https://github.com/mxxii/selderee}{\texttt{ selderee}}.\hypertarget{md_node__modules_2html-to-text_2_c_h_a_n_g_e_l_o_g_autotoc_md17780}{}\doxysubsection{\texorpdfstring{Base elements}{Base elements}}\label{md_node__modules_2html-to-text_2_c_h_a_n_g_e_l_o_g_autotoc_md17780}
The same selectors implementation is used now to narrow down the conversion to specific HTML DOM fragments. Addresses \href{https://github.com/html-to-text/node-html-to-text/issues/96}{\texttt{ \#96}}. (Previous implementation had more limited selectors format.)

BREAKING CHANGE\+: All outermost elements matching provided selectors will be present in the output (previously it was only the first match for each selector). Addresses \href{https://github.com/html-to-text/node-html-to-text/issues/215}{\texttt{ \#215}}.

{\ttfamily limits.\+max\+Base\+Elements} can be used when you only need a fixed number of base elements and would like to avoid checking the rest of the source HTML document.

Base elements can be arranged in output text in the order of matched selectors (default, to keep it closer to the old implementation) or in the order of appearance in source HTML document.

BREAKING CHANGE\+: previous implementation was treating id selectors in the same way as class selectors (could match {\ttfamily \texorpdfstring{$<$}{<}foo id="{}a b"{}\texorpdfstring{$>$}{>}} with {\ttfamily foo\#a} selector). New implementation is closer to the spec and doesn\textquotesingle{}t expect multiple ids on an element. You can achieve the old behavior with {\ttfamily foo\mbox{[}id\texorpdfstring{$\sim$}{\string~}=a\mbox{]}} selector in case you rely on it for some poorly formatted documents (note that it has different specificity though).\hypertarget{md_node__modules_2html-to-text_2_c_h_a_n_g_e_l_o_g_autotoc_md17781}{}\doxysubsection{\texorpdfstring{Batch processing}{Batch processing}}\label{md_node__modules_2html-to-text_2_c_h_a_n_g_e_l_o_g_autotoc_md17781}
Since options preprocessing is getting more involved with selectors compilation, it seemed reasonable to break the single {\ttfamily html\+To\+Text()} function into compilation and convertation steps. It might provide some performance benefits in client code.
