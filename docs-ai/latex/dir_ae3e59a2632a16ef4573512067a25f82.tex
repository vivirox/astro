\doxysubsection{Detailed Description}
\begin{quote}
Universal library to retrieve Git\+Hub’s identity URL for the OAuth web flow \end{quote}


\href{https://www.npmjs.com/package/@octokit/oauth-authorization-url}{\texttt{ }} \href{https://github.com/octokit/oauth-authorization-url.js/actions?query=workflow\%3ATest+branch\%3Amain}{\texttt{ }}

See \href{https://developer.github.com/apps/building-oauth-apps/authorizing-oauth-apps/\#web-application-flow}{\texttt{ Git\+Hub’s Developer Guide for the OAuth App web application flow}}. Note that the \href{https://docs.github.com/en/developers/apps/identifying-and-authorizing-users-for-github-apps\#web-application-flow}{\texttt{ OAuth web application flow for Git\+Hub Apps}} is slightly different. Git\+Hub Apps do not support scopes for its user access tokens (they are called user-\/to-\/server tokens for Git\+Hub Apps), instead they inherit the user permissions from the Git\+Hub App\textquotesingle{}s registration and the repository/organization access and permissions from the respective installation.

\cellcolor{\tableheadbgcolor}\textbf{ {\ttfamily scopes}  }&

Only relevant when {\ttfamily client\+Type} is set to {\ttfamily "{}oauth-\/app"{}}.

An array of scope names (or\+: space-\/delimited list of scopes). If not provided, scope defaults to an empty list for users that have not authorized any scopes for the application. For users who have authorized scopes for the application, the user won\textquotesingle{}t be shown the OAuth authorization page with the list of scopes. Instead, this step of the flow will automatically complete with the set of scopes the user has authorized for the application. For example, if a user has already performed the web flow twice and has authorized one token with user scope and another token with repo scope, a third web flow that does not provide a scope will receive a token with user and repo scope.
