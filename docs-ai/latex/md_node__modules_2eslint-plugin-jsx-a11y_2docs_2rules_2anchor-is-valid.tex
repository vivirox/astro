The HTML {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} element, with a valid {\ttfamily href} attribute, is formally defined as representing a {\bfseries{hyperlink}}. That is, a link between one HTML document and another, or between one location inside an HTML document and another location inside the same document.

In fact, the interactive, underlined {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} element has become so synonymous with web navigation that this expectation has become entrenched inside browsers, assistive technologies such as screen readers and in how people generally expect the internet to behave. In short, anchors should navigate.

The use of Java\+Script frameworks and libraries, like {\itshape React}, has made it very easy to add or subtract functionality from the standard HTML elements. This has led to {\itshape anchors} often being used in applications based on how they look and function instead of what they represent.

Whilst it is possible, for example, to turn the {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} element into a fully functional {\ttfamily \texorpdfstring{$<$}{<}button\texorpdfstring{$>$}{>}} element with ARIA, the native user agent implementations of HTML elements are to be preferred over custom ARIA solutions.\hypertarget{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2anchor-is-valid_autotoc_md13739}{}\doxysection{\texorpdfstring{How do I resolve this error?}{How do I resolve this error?}}\label{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2anchor-is-valid_autotoc_md13739}
\hypertarget{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2anchor-is-valid_autotoc_md13740}{}\doxysubsection{\texorpdfstring{Case\+: I want to perform an action and need a clickable UI element}{Case\+: I want to perform an action and need a clickable UI element}}\label{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2anchor-is-valid_autotoc_md13740}
The native user agent implementations of the {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} and {\ttfamily \texorpdfstring{$<$}{<}button\texorpdfstring{$>$}{>}} elements not only differ in how they look and how they act when activated, but also in how the user is expected to interact with them. Both are perfectly clickable when using a mouse, but keyboard users expect {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} to activate on {\ttfamily enter} only and {\ttfamily \texorpdfstring{$<$}{<}button\texorpdfstring{$>$}{>}} to activate on {\itshape both} {\ttfamily enter} and {\ttfamily space}.

This is exacerbated by the expectation sighted users have of how {\itshape buttons} and {\itshape anchors} work based on their appearance. Therefore we find that using {\itshape anchors} as {\itshape buttons} can easily create confusion without a relatively complicated ARIA and CSS implementation that only serves to create an element HTML already offers and browsers already implement fully accessibly.

We are aware that sometimes {\itshape anchors} are used instead of {\itshape buttons} to achieve a specific visual design. When using the {\ttfamily \texorpdfstring{$<$}{<}button\texorpdfstring{$>$}{>}} element this can still be achieved with styling but, due to the meaning many people attach to the standard underlined {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} due its appearance, please reconsider this in the design.

Consider the following\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{<a\ href="{}javascript:void(0)"{}\ onClick=\{foo\}>Perform\ action</a>}
\DoxyCodeLine{<a\ href="{}\#"{}\ onClick=\{foo\}>Perform\ action</a>}
\DoxyCodeLine{<a\ onClick=\{foo\}>Perform\ action</a>}

\end{DoxyCode}


All these {\itshape anchor} implementations indicate that the element is only used to execute Java\+Script code. All the above should be replaced with\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{<button\ onClick=\{foo\}>Perform\ action</button>}

\end{DoxyCode}
\hypertarget{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2anchor-is-valid_autotoc_md13741}{}\doxysubsection{\texorpdfstring{Case\+: I want navigable links}{Case\+: I want navigable links}}\label{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2anchor-is-valid_autotoc_md13741}
An {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} element without an {\ttfamily href} attribute no longer functions as a hyperlink. That means that it can no longer accept keyboard focus or be clicked on. The documentation for \doxysectlink{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2no-noninteractive-tabindex}{no-\/noninteractive-\/tabindex}{0} explores this further. Preferably use another element (such as {\ttfamily div} or {\ttfamily span}) for display of text.

To properly function as a hyperlink, the {\ttfamily href} attribute should be present and also contain a valid {\itshape URL}. {\itshape Java\+Script} strings, empty values or using only \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\#\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} are not considered valid {\ttfamily href} values.

Valid {\ttfamily href} attributes values are\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{<a\ href="{}/some/valid/uri"{}>Navigate\ to\ page</a>}
\DoxyCodeLine{<a\ href="{}/some/valid/uri\#top"{}>Navigate\ to\ page\ and\ location</a>}
\DoxyCodeLine{<a\ href="{}\#top"{}>Navigate\ to\ internal\ page\ location</a>}

\end{DoxyCode}
