You do not have to use the return\+Fatal\+Error function. Fatal errors will be returned in the normal error function in that case.

And if you want to return buffers instead of strings, you can do this by adding the {\ttfamily return\+Buffers} option.

If you handle with big numbers that are to large for JS (Number.\+MAX\+\_\+\+SAFE\+\_\+\+INTEGER === 2\texorpdfstring{$^\wedge$}{\string^}53 -\/ 16) please use the {\ttfamily string\+Numbers} option. That way all numbers are going to be returned as String and you can handle them safely.

\hypertarget{README.md_autotoc_md28068}{}\doxysubsubsection{\texorpdfstring{Protocol errors}{Protocol errors}}\label{README.md_autotoc_md28068}
To handle protocol errors (this is very unlikely to happen) gracefully you should add the return\+Fatal\+Error option, reject any still running command (they might have been processed properly but the reply is just wrong), destroy the socket and reconnect. Note that while doing this no new command may be added, so all new commands have to be buffered in the meantime, otherwise a chunk might still contain partial data of a following command that was already processed properly but answered in the same chunk as the command that resulted in the protocol error.\hypertarget{README.md_autotoc_md28069}{}\doxysubsubsection{\texorpdfstring{Contribute}{Contribute}}\label{README.md_autotoc_md28069}
