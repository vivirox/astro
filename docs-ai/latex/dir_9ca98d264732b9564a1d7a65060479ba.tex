You want to write maintainable tests for your Web UI. As a part of this goal, you want your tests to avoid including implementation details of your components and rather focus on making your tests give you the confidence for which they are intended. As part of this, you want your testbase to be maintainable in the long run so refactors of your components (changes to implementation but not functionality) don\textquotesingle{}t break your tests and slow you and your team down.\hypertarget{README.md_autotoc_md6277}{}\doxysubsubsection{\texorpdfstring{This Solution}{This Solution}}\label{README.md_autotoc_md6277}
The {\ttfamily DOM Testing Library} is a very light-\/weight solution for testing DOM nodes (whether simulated with \href{https://github.com/jsdom/jsdom}{\texttt{ {\ttfamily JSDOM}}} as provided by default with \href{https://facebook.github.io/jest}{\texttt{ Jest}} or in the browser). The main utilities it provides involve querying the DOM for nodes in a way that\textquotesingle{}s similar to how the user finds elements on the page. In this way, the library helps ensure your tests give you confidence in your UI code. The {\ttfamily DOM Testing Library}\textquotesingle{}s primary guiding principle is\+:

\begin{quote}
\href{https://twitter.com/kentcdodds/status/977018512689455106}{\texttt{ The more your tests resemble the way your software is used, the more confidence they can give you.}} \end{quote}

\begin{quote}
\href{https://twitter.com/kentcdodds/status/977018512689455106}{\texttt{ The more your tests resemble the way your software is used, the more confidence they can give you.}} \end{quote}


We try to only expose methods and utilities that encourage you to write tests that closely resemble how your web pages are used.

Utilities are included in this project based on the following guiding principles\+:


\begin{DoxyEnumerate}
\item If it relates to rendering components, it deals with DOM nodes rather than component instances, nor should it encourage dealing with component instances.
\item It should be generally useful for testing the application components in the way the user would use it. We {\itshape are} making some trade-\/offs here because we\textquotesingle{}re using a computer and often a simulated browser environment, but in general, utilities should encourage tests that use the components the way they\textquotesingle{}re intended to be used.
\item Utility implementations and APIs should be simple and flexible.
\end{DoxyEnumerate}

At the end of the day, what we want is for this library to be pretty light-\/weight, simple, and understandable.\hypertarget{README.md_autotoc_md6281}{}\doxysubsubsection{\texorpdfstring{Contributors}{Contributors}}\label{README.md_autotoc_md6281}
