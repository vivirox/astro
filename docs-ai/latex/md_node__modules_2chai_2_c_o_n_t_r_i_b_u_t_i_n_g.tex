We like to encourage you to contribute to the Chai.\+js repository. This should be as easy as possible for you but there are a few things to consider when contributing. The following guidelines for contribution should be followed if you want to submit a pull request or open an issue.

Following these guidelines helps to communicate that you respect the time of the developers managing and developing this open source project. In return, they should reciprocate that respect in addressing your issue or assessing patches and features.\hypertarget{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9848}{}\doxysubsubsection{\texorpdfstring{Table of Contents}{Table of Contents}}\label{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9848}

\begin{DoxyItemize}
\item Creating an Issue or Pull Request requires a \href{http://github.com}{\texttt{ Git\+Hub}} account.
\item Issue reports should be {\bfseries{clear}}, {\bfseries{concise}} and {\bfseries{reproducible}}. Check to see if your issue has already been resolved in the \mbox{[}master\mbox{]}() branch or already reported in Chai\textquotesingle{}s \href{https://github.com/chaijs/chai/issues}{\texttt{ Git\+Hub Issue Tracker}}.
\item Pull Requests must adhere to strict \href{https://github.com/chaijs/chai/wiki/Chai-Coding-Style-Guide}{\texttt{ coding style guidelines}}.
\item In general, avoid submitting PRs for new Assertions without asking core contributors first. More than likely it would be better implemented as a plugin.
\item Additional support is available via the \href{http://groups.google.com/group/chaijs}{\texttt{ Google Group}} or on irc.\+freenode.\+net\#chaijs.
\item {\bfseries{IMPORTANT}}\+: By submitting a patch, you agree to allow the project owner to license your work under the same license as that used by the project.
\end{DoxyItemize}

The issue tracker is the preferred channel for \doxylink{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_bugs}{bug reports}, \doxylink{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_features}{feature requests} and \doxylink{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_pull-requests}{submitting pull requests}, but please respect the following restrictions\+:


\begin{DoxyItemize}
\item Please {\bfseries{do not}} use the issue tracker for personal support requests (use \href{https://groups.google.com/forum/\#!forum/chaijs}{\texttt{ Google Group}} or IRC).
\item Please {\bfseries{do not}} derail or troll issues. Keep the discussion on topic and respect the opinions of others
\end{DoxyItemize}

 \hypertarget{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9851}{}\doxysubsection{\texorpdfstring{Bug Reports}{Bug Reports}}\label{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9851}
A bug is a {\bfseries{demonstrable problem}} that is caused by the code in the repository.

Guidelines for bug reports\+:


\begin{DoxyEnumerate}
\item {\bfseries{Use the Git\+Hub issue search}} --- check if the issue has already been reported.
\item {\bfseries{Check if the issue has been fixed}} --- try to reproduce it using the latest {\ttfamily master} or development branch in the repository.
\item {\bfseries{Isolate the problem}} --- create a test case to demonstrate your issue. Provide either a repo, gist, or code sample to demonstrate you problem.
\end{DoxyEnumerate}

A good bug report shouldn\textquotesingle{}t leave others needing to chase you up for more information. Please try to be as detailed as possible in your report. What is your environment? What steps will reproduce the issue? What browser(s) and/or Node.\+js versions experience the problem? What would you expect to be the outcome? All these details will help people to fix any potential bugs.

Example\+:

\begin{quote}
Short and descriptive example bug report title

A summary of the issue and the browser/\+OS environment in which it occurs. If suitable, include the steps required to reproduce the bug.


\begin{DoxyEnumerate}
\item This is the first step
\item This is the second step
\item Further steps, etc.
\end{DoxyEnumerate}

{\ttfamily \texorpdfstring{$<$}{<}url\texorpdfstring{$>$}{>}} -\/ a link to the reduced test case OR 
\begin{DoxyCode}{0}
\DoxyCodeLine{expect(a).to.equal('a');}
\DoxyCodeLine{//\ code\ sample}

\end{DoxyCode}


Any other information you want to share that is relevant to the issue being reported. This might include the lines of code that you have identified as causing the bug, and potential solutions (and your opinions on their merits). \end{quote}

 \hypertarget{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9852}{}\doxysubsection{\texorpdfstring{Feature Requests}{Feature Requests}}\label{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9852}
Feature requests are welcome. But take a moment to find out whether your idea fits with the scope and aims of the project. It\textquotesingle{}s up to {\itshape you} to make a strong case to convince the project\textquotesingle{}s developers of the merits of this feature. Please provide as much detail and context as possible.

Furthermore, since Chai.\+js has a \href{http://chaijs.com/guide/plugins/}{\texttt{ robust plugin API}}, we encourage you to publish {\bfseries{new Assertions}} as plugins. If your feature is an enhancement to an {\bfseries{existing Assertion}}, please propose your changes as an issue prior to opening a pull request. If the core Chai.\+js contributors feel your plugin would be better suited as a core assertion, they will invite you to open a PR in \href{https://github.com/chaijs/chai}{\texttt{ chaijs/chai}}.

\label{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_pull-requests}%
\Hypertarget{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_pull-requests}%
 \hypertarget{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9853}{}\doxysubsection{\texorpdfstring{Pull Requests}{Pull Requests}}\label{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9853}

\begin{DoxyItemize}
\item PRs for new core-\/assertions are advised against.
\item PRs for core-\/assertion bug fixes are always welcome.
\item PRs for enhancing the interfaces are always welcome.
\item PRs that increase test coverage are always welcome.
\item PRs are scrutinized for coding-\/style.
\end{DoxyItemize}

Good pull requests -\/ patches, improvements, new features -\/ are a fantastic help. They should remain focused in scope and avoid containing unrelated commits.

{\bfseries{Please ask first}} before embarking on any significant pull request (e.\+g. implementing features, refactoring code), otherwise you risk spending a lot of time working on something that the project\textquotesingle{}s developers might not want to merge into the project.

Please adhere to the coding conventions used throughout a project (indentation, accurate comments, etc.) and any other requirements (such as test coverage). Please review the \href{https://github.com/chaijs/chai/wiki/Chai-Coding-Style-Guide}{\texttt{ Chai.\+js Coding Style Guide}}.

Follow this process if you\textquotesingle{}d like your work considered for inclusion in the project\+:


\begin{DoxyEnumerate}
\item \href{http://help.github.com/fork-a-repo/}{\texttt{ Fork}} the project, clone your fork, and configure the remotes\+:
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{\#\ Clone\ your\ fork\ of\ the\ repo\ into\ the\ current\ directory}
\DoxyCodeLine{git\ clone\ https://github.com/<your-\/username>/<repo-\/name>}
\DoxyCodeLine{\#\ Navigate\ to\ the\ newly\ cloned\ directory}
\DoxyCodeLine{cd\ <repo-\/name>}
\DoxyCodeLine{\#\ Assign\ the\ original\ repo\ to\ a\ remote\ called\ "{}upstream"{}}
\DoxyCodeLine{git\ remote\ add\ upstream\ https://github.com/<upstream-\/owner>/<repo-\/name>}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item If you cloned a while ago, get the latest changes from upstream\+:
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ checkout\ <dev-\/branch>}
\DoxyCodeLine{git\ pull\ upstream\ <dev-\/branch>}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item Create a new topic branch (off the main project development branch) to contain your feature, change, or fix\+:
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ checkout\ -\/b\ <topic-\/branch-\/name>}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item Commit your changes in logical chunks. Use Git\textquotesingle{}s \href{https://help.github.com/articles/interactive-rebase}{\texttt{ interactive rebase}} feature to tidy up your commits before making them public.
\item Run you code to make sure it works. If you\textquotesingle{}re still having problems please try to run {\ttfamily make clean} and then test your code again.
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{npm\ test}
\DoxyCodeLine{\#\ when\ finished\ running\ tests...}
\DoxyCodeLine{git\ checkout\ chai.js}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item Locally merge (or rebase) the upstream development branch into your topic branch\+:
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ pull\ [-\/-\/rebase]\ upstream\ <dev-\/branch>}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item Push your topic branch up to your fork\+:
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ push\ origin\ <topic-\/branch-\/name>}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item \href{https://help.github.com/articles/using-pull-requests/}{\texttt{ Open a Pull Request}} with a clear title and description.
\end{DoxyEnumerate}

 \hypertarget{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9854}{}\doxysection{\texorpdfstring{Releasing}{Releasing}}\label{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9854}
Releases can be {\bfseries{prepared}} by anyone with access to the code.

Simply run {\ttfamily make release-\/major}, {\ttfamily make release-\/minor}, or {\ttfamily make-\/release-\/patch} and it will automatically do the following\+:


\begin{DoxyItemize}
\item Build chai.\+js
\item Bump the version numbers across the project
\item Make a commit within git
\end{DoxyItemize}

All you need to do is push the commit up and make a pull request, one of the core contributors will merge it and publish a release.\hypertarget{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9855}{}\doxysubsection{\texorpdfstring{Publishing a Release}{Publishing a Release}}\label{md_node__modules_2chai_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md9855}
Anyone who is a core contributor (see the \href{https://github.com/chaijs/chai\#core-contributors}{\texttt{ Core Contributors Heading in the Readme}}) can publish a release\+:


\begin{DoxyEnumerate}
\item Go to the \href{https://github.com/chaijs/chai/releases}{\texttt{ Releases page on Github}}
\item Hit "{}\+Draft a new release"{} (if you can\textquotesingle{}t see this, you\textquotesingle{}re not a core contributor!)
\item Write human-\/friendly Release Notes based on changelog.
\begin{DoxyItemize}
\item The release title is "{}x.\+x.\+x / YYYY-\/\+MM-\/\+DD"{} (where x.\+x.\+x is the version number)
\item If breaking changes, write migration tutorial(s) and reasoning.
\item Callouts for community contributions (PRs) with links to PR and contributing user.
\item Callouts for other fixes made by core contributors with links to issue.
\end{DoxyItemize}
\item Hit "{}\+Save Draft"{} and get other core contributors to check your work, or alternatively hit "{}\+Publish release"{}
\item That\textquotesingle{}s it!
\end{DoxyEnumerate}
