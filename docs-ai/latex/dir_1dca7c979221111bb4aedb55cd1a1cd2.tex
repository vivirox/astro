Simple, transparent parser combinators toolkit that supports tokens of any type.

For when you wanna do weird things with parsers.\hypertarget{README.md_autotoc_md25880}{}\doxysubsection{\texorpdfstring{Features}{Features}}\label{README.md_autotoc_md25880}

\begin{DoxyItemize}
\item {\bfseries{Well typed}} -\/ written in Type\+Script and with a lot of attention to keep types well defined.
\item {\bfseries{Highly generic}} -\/ no constraints on tokens, options (additional state data) and output types. Core module has not a single mention of strings as a part of normal flow. Some string-\/specific building blocks can be loaded from a separate module in case you need them.
\item {\bfseries{Transparent}}. Built on a very simple base idea -\/ just a few type aliases. Whole parser state is accessible at any time.
\item {\bfseries{Lightweight}}. Zero dependencies. Just type aliases and functions.
\item {\bfseries{Batteries included}} -\/ comes with a pretty big set of building blocks.
\item {\bfseries{Easy to extend}} -\/ just follow the convention defined by type aliases when making your own building blocks. {\itshape (And maybe let me know what you think can be universally useful to be included in the package itself.)}
\item {\bfseries{Easy to make configurable parsers}}. Rather than dynamically composing parsers based on options or manually weaving options into a dynamic parser state, this package offers a standard way to treat options as a part of static data and access them at any moment for course correction.
\item {\bfseries{Well tested}} -\/ comes with tests for everything including examples.
\item {\bfseries{Practicality over "{}purity"{}}}. To be understandable and self-\/consistent is more important than to follow an established encoding of abstract ideas. More on this below.
\item {\bfseries{No streaming}} -\/ accepts a fixed array of tokens. It is simple, whole input can be accessed at any time if needed. More on this below.
\item {\bfseries{Bring your own lexer/tokenizer}} -\/ if you need it. It doesn\textquotesingle{}t matter how tokens are made -\/ this package can consume anything you can type. I have a lexer as well, called \href{https://github.com/mxxii/leac}{\texttt{ leac}}, and it is used in some examples, but there is nothing special about it to make it the best match (well, maybe the fact it is written in Type\+Script, has equal level of maintenance and is made with arrays instead of iterators in mind as well).
\end{DoxyItemize}\hypertarget{README.md_autotoc_md25881}{}\doxysubsection{\texorpdfstring{Changelog}{Changelog}}\label{README.md_autotoc_md25881}

\begin{DoxyItemize}
\item {\bfseries{Data}} object holds tokens array and possibly an options object -\/ it\textquotesingle{}s just a container for all static data used by a parser. Parser position, on the other hand, has it\textquotesingle{}s own life cycle and passed around separately.
\item A {\bfseries{Parser}} is a function that accepts Data object and a parser position, looks into the tokens array at the given position and returns either a Match with a parsed value (use {\ttfamily null} if there is no value) and a new position or a Non\+Match.
\item A {\bfseries{Matcher}} is a special case of Parser that never fails and always returns a Match.
\item {\bfseries{Result}} object from a Parser can be either a Match or a Non\+Match.
\item {\bfseries{Match}} is a result of successful parsing -\/ it contains a parsed value and a new parser position.
\item {\bfseries{Non\+Match}} is a result of unsuccessful parsing. It doesn\textquotesingle{}t have any data attached to it.
\item {\bfseries{TToken}} can be any type.
\item {\bfseries{TOptions}} can be any type. Use it to make your parser customizable. Or set it as {\ttfamily undefined} and type as {\ttfamily unknown} if not needed.
\end{DoxyItemize}\hypertarget{README.md_autotoc_md25889}{}\doxysubsubsection{\texorpdfstring{Building blocks}{Building blocks}}\label{README.md_autotoc_md25889}

\hypertarget{README.md_autotoc_md25894}{}\doxysubsection{\texorpdfstring{What about ...?}{What about ...?}}\label{README.md_autotoc_md25894}

\begin{DoxyItemize}
\item performance -\/ The code is very simple but I won\textquotesingle{}t put any unverified assumptions here. I\textquotesingle{}d be grateful to anyone who can set up a good benchmark project to compare different parser combinators.
\item stable release -\/ Current release is well thought out and tested. I leave a chance that some supplied functions may need an incompatible change. Before version 1.\+0.\+0 this will be done without a deprecation cycle.
\item streams/iterators -\/ Maybe some day, if the need to parse a stream of non-\/string data arise. For now I don\textquotesingle{}t have a task that would force me to think well on how to design it. It would require a significant trade off and may end up being a separate module (like {\ttfamily char}) at best or even a separate package.
\item Fantasy Land -\/ You can find some familiar ideas here, especially when compared to Static Land. But I\textquotesingle{}m not concerned about compatibility with that spec -\/ see "{}\+Practicality over "{}purity"{}"{} entry above. What I think might make sense is to add separate tests for laws applicable in context of this package. Low priority though.
\end{DoxyItemize}\hypertarget{README.md_autotoc_md25895}{}\doxysubsection{\texorpdfstring{Some other parser combinator packages}{Some other parser combinator packages}}\label{README.md_autotoc_md25895}
