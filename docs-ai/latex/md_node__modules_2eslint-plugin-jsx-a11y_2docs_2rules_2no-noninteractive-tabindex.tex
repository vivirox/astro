Tab key navigation should be limited to elements on the page that can be interacted with. Thus it is not necessary to add a tabindex to items in an unordered list, for example, to make them navigable through assistive technology. These applications already afford page traversal mechanisms based on the HTML of the page. Generally, we should try to reduce the size of the page\textquotesingle{}s tab ring rather than increasing it.\hypertarget{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2no-noninteractive-tabindex_autotoc_md13924}{}\doxysection{\texorpdfstring{How do I resolve this error?}{How do I resolve this error?}}\label{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2no-noninteractive-tabindex_autotoc_md13924}

The {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} tag is tricky. Consider the following\+:

The bare {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} tag is an {\itshape anchor}. It has no semantic AX API mapping in either ARIA or the AXObject model. It\textquotesingle{}s as meaningful as {\ttfamily \texorpdfstring{$<$}{<}div\texorpdfstring{$>$}{>}}, which is to say it has no meaning. An {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} tag with an {\ttfamily href} attribute has an inherent role of {\ttfamily link}. An {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}} tag with an explicit role obtains the designated role. In the example above, this role is {\ttfamily button}.\hypertarget{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2no-noninteractive-tabindex_autotoc_md13926}{}\doxysubsection{\texorpdfstring{Case\+: I am using "{}semantic"{} HTML. Isn\textquotesingle{}t that interactive?}{Case\+: I am using "{}semantic"{} HTML. Isn\textquotesingle{}t that interactive?}}\label{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2no-noninteractive-tabindex_autotoc_md13926}
If we take a step back into the field of linguistics for a moment, let\textquotesingle{}s consider what it means for something to be "{}semantic"{}. Nothing, in and of itself, has meaning. Meaning is constructed through dialogue. A speaker intends a meaning and a listener/observer interprets a meaning. Each participant constructs their own meaning through dialogue. There is no intrinsic or isolated meaning outside of interaction. Thus, we must ask, given that we have a "{}speaker"{} who communicates via "{}semantic"{} HTML, who is listening/observing?

In our case, the observer is the Accessibility (AX) API. Browsers interpret HTML (inflected at times by ARIA) to construct a meaning (AX Tree) of the page. Whatever the semantic HTML intends has only the force of suggestion to the AX API. Therefore, we have inconsistencies. For example, there is not yet an ARIA role for {\ttfamily text} or {\ttfamily label} and thus no way to change a {\ttfamily \texorpdfstring{$<$}{<}label\texorpdfstring{$>$}{>}} into plain text or a {\ttfamily \texorpdfstring{$<$}{<}span\texorpdfstring{$>$}{>}} into a label via ARIA. {\ttfamily \texorpdfstring{$<$}{<}div\texorpdfstring{$>$}{>}} has an AXObject correpondant {\ttfamily Div\+Role}, but no such object maps to {\ttfamily \texorpdfstring{$<$}{<}span\texorpdfstring{$>$}{>}}.

What this lint rule endeavors to do is apply the AX API understanding of the semantics of an HTML document back onto your code. The concept of interactivity boils down to whether a user can do something with the indicated or focused component.

Endeavor to limit tabbable elements to those that a user can act upon.\hypertarget{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2no-noninteractive-tabindex_autotoc_md13927}{}\doxysubsection{\texorpdfstring{Case\+: Shouldn\textquotesingle{}t I add a tabindex so that users can navigate to this item?}{Case\+: Shouldn\textquotesingle{}t I add a tabindex so that users can navigate to this item?}}\label{md_node__modules_2eslint-plugin-jsx-a11y_2docs_2rules_2no-noninteractive-tabindex_autotoc_md13927}
It is not necessary to put a tabindex on an {\ttfamily \texorpdfstring{$<$}{<}article\texorpdfstring{$>$}{>}}, for instance or on {\ttfamily \texorpdfstring{$<$}{<}li\texorpdfstring{$>$}{>}} items; assistive technologies provide affordances to users to find and traverse these containers. Most elements that require a tabindex -- {\ttfamily \texorpdfstring{$<$}{<}a href\texorpdfstring{$>$}{>}}, {\ttfamily \texorpdfstring{$<$}{<}button\texorpdfstring{$>$}{>}}, {\ttfamily \texorpdfstring{$<$}{<}input\texorpdfstring{$>$}{>}}, {\ttfamily \texorpdfstring{$<$}{<}textarea\texorpdfstring{$>$}{>}} -- have it already.

Your application might require an exception to this rule in the case of an element that captures incoming tab traversal for a composite widget. In that case, turn off this rule on a per instance basis. This is an uncommon case.

If you know that a particular element will be scrollable, you might want to add {\ttfamily tabindex="{}0"{}} if your website supports browsers that don\textquotesingle{}t make these containers keyboard-\/focusable. The current status for this platform feature can be tracked in \href{https://www.chromestatus.com/feature/5231964663578624}{\texttt{ Chrome Platform Status "{}\+Feature\+: Keyboard-\/focusable scroll containers"{}}}.
