To understand what this project does, it‚Äôs very important to first understand what unified does\+: please read through the \href{https://github.com/unifiedjs/unified}{\texttt{ {\ttfamily unifiedjs/unified}}} readme (the part until you hit the API section is required reading).

{\ttfamily @mdx-\/js/mdx} is a unified pipeline ‚Äî wrapped so that most folks don‚Äôt need to know about unified. The processor goes through these steps\+:


\begin{DoxyEnumerate}
\item parse MDX (serialized markdown with embedded JSX, ESM, and expressions) to mdast (markdown syntax tree)
\item transform through remark (markdown ecosystem)
\item transform mdast to hast (HTML syntax tree)
\item transform through rehype (HTML ecosystem)
\item transform hast to esast (JS syntax tree)
\item do the work needed to get a component
\item transform through recma (JS ecosystem)
\item serialize esast as Java\+Script
\end{DoxyEnumerate}

The {\itshape input} is MDX (serialized markdown with embedded JSX, ESM, and expressions). The markdown is parsed with \href{https://github.com/micromark/micromark}{\texttt{ {\ttfamily micromark/micromark}}} and the embedded JS with one of its extensions \href{https://github.com/micromark/micromark-extension-mdxjs}{\texttt{ {\ttfamily micromark/micromark-\/extension-\/mdxjs}}} (which in turn uses \href{https://github.com/acornjs/acorn}{\texttt{ acorn}}). Then \href{https://github.com/syntax-tree/mdast-util-from-markdown}{\texttt{ {\ttfamily syntax-\/tree/mdast-\/util-\/from-\/markdown}}} and its extension \href{https://github.com/syntax-tree/mdast-util-mdx}{\texttt{ {\ttfamily syntax-\/tree/mdast-\/util-\/mdx}}} are used to turn the results from the parser into a syntax tree\+: \href{https://github.com/syntax-tree/mdast}{\texttt{ mdast}}.

Markdown is closest to the source format. This is where \href{https://github.com/remarkjs/remark/blob/main/doc/plugins.md\#list-of-plugins}{\texttt{ remark plugins}} come in. Typically, there shouldn‚Äôt be much going on here. But perhaps you want to support GFM (tables and such) or frontmatter? Then you can add a plugin here\+: {\ttfamily remark-\/gfm} or {\ttfamily remark-\/frontmatter}, respectively.

After markdown, we go to \href{https://github.com/syntax-tree/hast}{\texttt{ hast}} (HTML). This transformation is done by \href{https://github.com/syntax-tree/mdast-util-to-hast}{\texttt{ {\ttfamily syntax-\/tree/mdast-\/util-\/to-\/hast}}}. Wait, why, what is HTML needed? Part of the reason is that we care about HTML semantics\+: we want to know that something is an {\ttfamily \texorpdfstring{$<$}{<}a\texorpdfstring{$>$}{>}}, not whether it‚Äôs a link with a resource ({\ttfamily \mbox{[}text\mbox{]}(url)}) or a reference to a defined link definition ({\ttfamily \mbox{[}text\mbox{]}\mbox{[}id\mbox{]}\textbackslash{}n\textbackslash{}n\mbox{[}id\mbox{]}\+: url}). So an HTML AST is {\itshape closer} to where we want to go. Another reason is that there are many things folks need when they go MDX -\/\texorpdfstring{$>$}{>} JS, markdown -\/\texorpdfstring{$>$}{>} HTML, or even folks who only process their HTML -\/\texorpdfstring{$>$}{>} HTML\+: use cases other than MDX. By having a single AST in these cases and writing a plugin that works on that AST, that plugin can supports {\itshape all} these use cases (for example, \href{https://github.com/rehypejs/rehype-highlight}{\texttt{ {\ttfamily rehypejs/rehype-\/highlight}}} for syntax highlighting or \href{https://github.com/remarkjs/remark-math/tree/main/packages/rehype-katex}{\texttt{ {\ttfamily rehypejs/rehype-\/katex}}} for math). So, this is where \href{https://github.com/rehypejs/rehype/blob/main/doc/plugins.md\#list-of-plugins}{\texttt{ rehype plugins}} come in\+: most of the plugins, probably.

Then we go to Java\+Script\+: \href{https://github.com/syntax-tree/esast}{\texttt{ esast}} (JS; an AST which is compatible with estree but looks a bit more like other unist ASTs). This transformation is done by \href{https://github.com/mdx-js/recma/tree/main/packages/rehype-recma}{\texttt{ {\ttfamily rehype-\/recma}}}. This is a new ecosystem that does not have utilities or plugins yet. But it‚Äôs where {\ttfamily @mdx-\/js/mdx} does its thing\+: where it adds imports/exports, where it compiles JSX away into {\ttfamily \+\_\+jsx()} calls, and where it does the other cool things that it provides.

Finally, The output is serialized Java\+Script.
