\doxysubsection{Detailed Description}
Match files using the patterns the shell uses.

The most correct and second fastest glob implementation in Java\+Script. (See {\bfseries{Comparison to Other Java\+Script Glob Implementations}} at the bottom of this readme.)

Much more information about glob pattern expansion can be found by running {\ttfamily man bash} and searching for {\ttfamily Pattern Matching}.

"{}\+Globs"{} are the patterns you type when you do stuff like {\ttfamily ls \texorpdfstring{$\ast$}{*}.js} on the command line, or put {\ttfamily build/\texorpdfstring{$\ast$}{*}} in a {\ttfamily .gitignore} file.

Before parsing the path part patterns, braced sections are expanded into a set. Braced sections start with {\ttfamily \{} and end with {\ttfamily \}}, with 2 or more comma-\/delimited sections within. Braced sections may contain slash characters, so {\ttfamily a\{/b/c,bcd\}} would expand into {\ttfamily a/b/c} and {\ttfamily abcd}.

The following characters have special magic meaning when used in a path portion. With the exception of {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}, none of these match path separators (ie, {\ttfamily /} on all platforms, and {\ttfamily \textbackslash{}} on Windows).


\begin{DoxyItemize}
\item {\ttfamily \texorpdfstring{$\ast$}{*}} Matches 0 or more characters in a single path portion. When alone in a path portion, it must match at least 1 character. If {\ttfamily dot\+:true} is not specified, then {\ttfamily \texorpdfstring{$\ast$}{*}} will not match against a {\ttfamily .} character at the start of a path portion.
\item {\ttfamily ?} Matches 1 character. If {\ttfamily dot\+:true} is not specified, then {\ttfamily ?} will not match against a {\ttfamily .} character at the start of a path portion.
\item {\ttfamily \mbox{[}...\mbox{]}} Matches a range of characters, similar to a Reg\+Exp range. If the first character of the range is {\ttfamily !} or {\ttfamily \texorpdfstring{$^\wedge$}{\string^}} then it matches any character not in the range. If the first character is {\ttfamily \mbox{]}}, then it will be considered the same as {\ttfamily \textbackslash{}\mbox{]}}, rather than the end of the character class.
\item {\ttfamily !(pattern\texorpdfstring{$\vert$}{|}pattern\texorpdfstring{$\vert$}{|}pattern)} Matches anything that does not match any of the patterns provided. May {\itshape not} contain {\ttfamily /} characters. Similar to {\ttfamily \texorpdfstring{$\ast$}{*}}, if alone in a path portion, then the path portion must have at least one character.
\item {\ttfamily ?(pattern\texorpdfstring{$\vert$}{|}pattern\texorpdfstring{$\vert$}{|}pattern)} Matches zero or one occurrence of the patterns provided. May {\itshape not} contain {\ttfamily /} characters.
\item {\ttfamily +(pattern\texorpdfstring{$\vert$}{|}pattern\texorpdfstring{$\vert$}{|}pattern)} Matches one or more occurrences of the patterns provided. May {\itshape not} contain {\ttfamily /} characters.
\item {\ttfamily \texorpdfstring{$\ast$}{*}(a\texorpdfstring{$\vert$}{|}b\texorpdfstring{$\vert$}{|}c)} Matches zero or more occurrences of the patterns provided. May {\itshape not} contain {\ttfamily /} characters.
\item {\ttfamily @(pattern\texorpdfstring{$\vert$}{|}pat\texorpdfstring{$\ast$}{*}\texorpdfstring{$\vert$}{|}pat?erN)} Matches exactly one of the patterns provided. May {\itshape not} contain {\ttfamily /} characters.
\item {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} If a "{}globstar"{} is alone in a path portion, then it matches zero or more directories and subdirectories searching for matches. It does not crawl symlinked directories, unless {\ttfamily \{follow\+:true\}} is passed in the options object. A pattern like {\ttfamily a/b/\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} will only match {\ttfamily a/b} if it is a directory. Follows 1 symbolic link if not the first item in the pattern, or 0 if it is the first item, unless {\ttfamily follow\+:true} is set, in which case it follows all symbolic links.
\end{DoxyItemize}

If a file or directory path portion has a {\ttfamily .} as the first character, then it will not match any glob pattern unless that pattern\textquotesingle{}s corresponding path part also has a {\ttfamily .} as its first character.

For example, the pattern {\ttfamily a/.\texorpdfstring{$\ast$}{*}/c} would match the file at {\ttfamily a/.b/c}. However the pattern {\ttfamily a/\texorpdfstring{$\ast$}{*}/c} would not, because {\ttfamily \texorpdfstring{$\ast$}{*}} does not start with a dot character.

You can make glob treat dots as normal characters by setting {\ttfamily dot\+:true} in the options.\hypertarget{README.md_autotoc_md16226}{}\doxysubsubsection{\texorpdfstring{Basename Matching}{Basename Matching}}\label{README.md_autotoc_md16226}
If you set {\ttfamily match\+Base\+:true} in the options, and the pattern has no slashes in it, then it will seek for any file anywhere in the tree with a matching basename. For example, {\ttfamily \texorpdfstring{$\ast$}{*}.js} would match {\ttfamily test/simple/basic.\+js}.\hypertarget{README.md_autotoc_md16227}{}\doxysubsubsection{\texorpdfstring{Empty Sets}{Empty Sets}}\label{README.md_autotoc_md16227}
If no matching files are found, then an empty array is returned. This differs from the shell, where the pattern itself is returned. For example\+:

\hypertarget{README.md_autotoc_md16228}{}\doxysubsection{\texorpdfstring{Comparisons to other fnmatch/glob implementations}{Comparisons to other fnmatch/glob implementations}}\label{README.md_autotoc_md16228}
While strict compliance with the existing standards is a worthwhile goal, some discrepancies exist between node-\/glob and other implementations, and are intentional.

The double-\/star character {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} is supported by default, unless the {\ttfamily noglobstar} flag is set. This is supported in the manner of bsdglob and bash 5, where {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} only has special significance if it is the only thing in a path part. That is, {\ttfamily a/\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/b} will match {\ttfamily a/x/y/b}, but {\ttfamily a/\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}b} will not.

Note that symlinked directories are not traversed as part of a {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}, though their contents may match against subsequent portions of the pattern. This prevents infinite loops and duplicates and the like. You can force glob to traverse symlinks with {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} by setting {\ttfamily \{follow\+:true\}} in the options.

There is no equivalent of the {\ttfamily nonull} option. A pattern that does not find any matches simply resolves to nothing. (An empty array, immediately ended stream, etc.)

If brace expansion is not disabled, then it is performed before any other interpretation of the glob pattern. Thus, a pattern like {\ttfamily +(a\texorpdfstring{$\vert$}{|}\{b),c)\}}, which would not be valid in bash or zsh, is expanded {\bfseries{first}} into the set of {\ttfamily +(a\texorpdfstring{$\vert$}{|}b)} and {\ttfamily +(a\texorpdfstring{$\vert$}{|}c)}, and those patterns are checked for validity. Since those two are valid, matching proceeds.

The character class patterns {\ttfamily \mbox{[}\+:class\+:\mbox{]}} (posix standard named classes) style class patterns are supported and unicode-\/aware, but {\ttfamily \mbox{[}=c=\mbox{]}} (locale-\/specific character collation weight), and {\ttfamily \mbox{[}.symbol.\mbox{]}} (collating symbol), are not.\hypertarget{README.md_autotoc_md16229}{}\doxysubsubsection{\texorpdfstring{Repeated Slashes}{Repeated Slashes}}\label{README.md_autotoc_md16229}
Unlike Bash and zsh, repeated {\ttfamily /} are always coalesced into a single path separator.\hypertarget{README.md_autotoc_md16230}{}\doxysubsubsection{\texorpdfstring{Comments and Negation}{Comments and Negation}}\label{README.md_autotoc_md16230}
Previously, this module let you mark a pattern as a "{}comment"{} if it started with a {\ttfamily \#} character, or a "{}negated"{} pattern if it started with a {\ttfamily !} character.

These options were deprecated in version 5, and removed in version 6.

To specify things that should not match, use the {\ttfamily ignore} option.\hypertarget{README.md_autotoc_md16231}{}\doxysubsection{\texorpdfstring{Windows}{Windows}}\label{README.md_autotoc_md16231}
{\bfseries{Please only use forward-\/slashes in glob expressions.}}

Though windows uses either {\ttfamily /} or {\ttfamily \textbackslash{}} as its path separator, only {\ttfamily /} characters are used by this glob implementation. You must use forward-\/slashes {\bfseries{only}} in glob expressions. Back-\/slashes will always be interpreted as escape characters, not path separators.

Results from absolute patterns such as {\ttfamily /foo/\texorpdfstring{$\ast$}{*}} are mounted onto the root setting using {\ttfamily path.\+join}. On windows, this will by default result in {\ttfamily /foo/\texorpdfstring{$\ast$}{*}} matching {\ttfamily C\+:\textbackslash{}foo\textbackslash{}bar.\+txt}.

To automatically coerce all {\ttfamily \textbackslash{}} characters to {\ttfamily /} in pattern strings, {\bfseries{thus making it impossible to escape literal glob characters}}, you may set the {\ttfamily windows\+Paths\+No\+Escape} option to {\ttfamily true}.\hypertarget{README.md_autotoc_md16232}{}\doxysubsubsection{\texorpdfstring{Windows, CWDs, Drive Letters, and UNC Paths}{Windows, CWDs, Drive Letters, and UNC Paths}}\label{README.md_autotoc_md16232}
On posix systems, when a pattern starts with {\ttfamily /}, any {\ttfamily cwd} option is ignored, and the traversal starts at {\ttfamily /}, plus any non-\/magic path portions specified in the pattern.

On Windows systems, the behavior is similar, but the concept of an "{}absolute path"{} is somewhat more involved.\hypertarget{README.md_autotoc_md16233}{}\doxysubsubsubsection{\texorpdfstring{UNC Paths}{UNC Paths}}\label{README.md_autotoc_md16233}
A UNC path may be used as the start of a pattern on Windows platforms. For example, a pattern like\+: {\ttfamily //?/x\+:/\texorpdfstring{$\ast$}{*}} will return all file entries in the root of the {\ttfamily x\+:} drive. A pattern like {\ttfamily //\+Computer\+Name/\+Share/\texorpdfstring{$\ast$}{*}} will return all files in the associated share.

UNC path roots are always compared case insensitively.\hypertarget{README.md_autotoc_md16234}{}\doxysubsubsubsection{\texorpdfstring{Drive Letters}{Drive Letters}}\label{README.md_autotoc_md16234}
A pattern starting with a drive letter, like {\ttfamily c\+:/\texorpdfstring{$\ast$}{*}}, will search in that drive, regardless of any {\ttfamily cwd} option provided.

If the pattern starts with {\ttfamily /}, and is not a UNC path, and there is an explicit {\ttfamily cwd} option set with a drive letter, then the drive letter in the {\ttfamily cwd} is used as the root of the directory traversal.

For example, `glob('/tmp\textquotesingle{}, \{ cwd\+: \textquotesingle{}c\+:/any/thing\textquotesingle{} \}){\ttfamily will return }\mbox{[}\textquotesingle{}c\+:/tmp\textquotesingle{}\mbox{]}\`{} as the result.

If an explicit {\ttfamily cwd} option is not provided, and the pattern starts with {\ttfamily /}, then the traversal will run on the root of the drive provided as the {\ttfamily cwd} option. (That is, it is the result of `path.\+resolve('/\textquotesingle{})\`{}.)\hypertarget{README.md_autotoc_md16235}{}\doxysubsection{\texorpdfstring{Race Conditions}{Race Conditions}}\label{README.md_autotoc_md16235}
Glob searching, by its very nature, is susceptible to race conditions, since it relies on directory walking.

As a result, it is possible that a file that exists when glob looks for it may have been deleted or modified by the time it returns the result.

By design, this implementation caches all readdir calls that it makes, in order to cut down on system overhead. However, this also makes it even more susceptible to races, especially if the cache object is reused between glob calls.

Users are thus advised not to use a glob result as a guarantee of filesystem state in the face of rapid changes. For the vast majority of operations, this is never a problem.\hypertarget{README.md_autotoc_md16236}{}\doxysubsubsection{\texorpdfstring{See Also\+:}{See Also\+:}}\label{README.md_autotoc_md16236}

\hypertarget{README.md_autotoc_md16239}{}\doxysubsection{\texorpdfstring{Comparison to Other Java\+Script Glob Implementations}{Comparison to Other Java\+Script Glob Implementations}}\label{README.md_autotoc_md16239}
{\bfseries{tl;dr}}


\begin{DoxyItemize}
\item If you want glob matching that is as faithful as possible to Bash pattern expansion semantics, and as fast as possible within that constraint, {\itshape use this module}.
\item If you are reasonably sure that the patterns you will encounter are relatively simple, and want the absolutely fastest glob matcher out there, {\itshape use \href{http://npm.im/fast-glob}{\texttt{ fast-\/glob}}}.
\item If you are reasonably sure that the patterns you will encounter are relatively simple, and want the convenience of automatically respecting {\ttfamily .gitignore} files, {\itshape use \href{http://npm.im/globby}{\texttt{ globby}}}.
\end{DoxyItemize}

There are some other glob matcher libraries on npm, but these three are (in my opinion, as of 2023) the best.

\DoxyHorRuler{0}


{\bfseries{full explanation}}

Every library reflects a set of opinions and priorities in the trade-\/offs it makes. Other than this library, I can personally recommend both \href{http://npm.im/globby}{\texttt{ globby}} and \href{http://npm.im/fast-glob}{\texttt{ fast-\/glob}}, though they differ in their benefits and drawbacks.

Both have very nice APIs and are reasonably fast.

{\ttfamily fast-\/glob} is, as far as I am aware, the fastest glob implementation in Java\+Script today. However, there are many cases where the choices that {\ttfamily fast-\/glob} makes in pursuit of speed mean that its results differ from the results returned by Bash and other sh-\/like shells, which may be surprising.

In my testing, {\ttfamily fast-\/glob} is around 10-\/20\% faster than this module when walking over 200k files nested 4 directories deep1. However, there are some inconsistencies with Bash matching behavior that this module does not suffer from\+:


\begin{DoxyItemize}
\item {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} only matches files, not directories
\item {\ttfamily ..} path portions are not handled unless they appear at the start of the pattern
\item {\ttfamily ./!(\texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>})} will not match any files that {\itshape start} with {\ttfamily \texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>}}, even if they do not match {\ttfamily \texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>}}. For example, {\ttfamily !(9).txt} will not match {\ttfamily 9999.\+txt}.
\item Some brace patterns in the middle of a pattern will result in failing to find certain matches.
\item Extglob patterns are allowed to contain {\ttfamily /} characters.
\end{DoxyItemize}

Globby exhibits all of the same pattern semantics as fast-\/glob, (as it is a wrapper around fast-\/glob) and is slightly slower than node-\/glob (by about 10-\/20\% in the benchmark test set, or in other words, anywhere from 20-\/50\% slower than fast-\/glob). However, it adds some API conveniences that may be worth the costs.


\begin{DoxyItemize}
\item Support for {\ttfamily .gitignore} and other ignore files.
\item Support for negated globs (ie, patterns starting with {\ttfamily !} rather than using a separate {\ttfamily ignore} option).
\end{DoxyItemize}

The priority of this module is "{}correctness"{} in the sense of performing a glob pattern expansion as faithfully as possible to the behavior of Bash and other sh-\/like shells, with as much speed as possible.

Note that prior versions of {\ttfamily node-\/glob} are {\itshape not} on this list. Former versions of this module are far too slow for any cases where performance matters at all, and were designed with APIs that are extremely dated by current Java\+Script standards.
